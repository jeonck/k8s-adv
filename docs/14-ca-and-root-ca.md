# 인증서 발급 기관: Root CA 와 CA

Kubernetes API Server 인증서 이해를 위한 인증서 발급 기관 (CA) 에 대해 알아봅니다.

---

## 공개키 배포와 암호화 과정

### 1. 수신자는 공개키를 공개하는가?

```
┌─────────────────────────────────────────────────────────────┐
│              공개키 배포 방식                               │
└─────────────────────────────────────────────────────────────┘

네, 공개키 (Public Key) 는 말 그대로 누구에게나 알려져도 됩니다.

웹사이트의 경우:
  ┌─────────────────────────────────────────────────────────┐
  │  웹 서버 (수신자) 는 접속하려는 모든 사용자 (발신자) 에게 │
  │  자신의 공개키를 담은 인증서를 미리 배포하거나          │
  │  요청 시 즉시 전달합니다.                               │
  └─────────────────────────────────────────────────────────┘

개인의 경우:
  ┌─────────────────────────────────────────────────────────┐
  │  자신의 공개키를 키 서버 (Key Server) 에 등록하거나     │
  │  이메일 서명 등에 첨부하여                              │
  │  상대방이 언제든 가져가서 사용할 수 있게 합니다.        │
  └─────────────────────────────────────────────────────────┘
```

### 2. 암호화 → 복호화 과정

```
┌─────────────────────────────────────────────────────────────┐
│              비대칭키 암호화 과정                           │
└─────────────────────────────────────────────────────────────┘

Step 1: 키 생성 (수신자)
  ┌─────────────────────────────────────────┐
  │  수신자가 수학적으로 연결된 키 쌍 생성 │
  │  - 공개키 (Public Key)                  │
  │  - 개인키 (Private Key)                 │
  └─────────────────────────────────────────┘

Step 2: 공개 (수신자)
  ┌─────────────────────────────────────────┐
  │  공개키 → 세상에 공개                   │
  │  개인키 → 자신만 안전하게 보관          │
  └─────────────────────────────────────────┘

Step 3: 암호화 (발신자)
  ┌─────────────────────────────────────────┐
  │  발신자가 수신자의 공개키를 획득        │
  │  보낼 데이터를 공개키로 암호화          │
  └─────────────────────────────────────────┘

Step 4: 전송
  ┌─────────────────────────────────────────┐
  │  암호화된 데이터 전송                   │
  │  중간 탈취 시 개인키 없이는 복호화 불가 │
  └─────────────────────────────────────────┘

Step 5: 복호화 (수신자)
  ┌─────────────────────────────────────────┐
  │  수신자가 개인키로 암호화된 데이터 복호 │
  │  원문 확인                              │
  └─────────────────────────────────────────┘
```

### 3. 하이브리드 방식: 대칭키 + 비대칭키

```
┌─────────────────────────────────────────────────────────────┐
│              실제 통신: 하이브리드 방식                     │
└─────────────────────────────────────────────────────────────┘

비대칭키만 사용하지 않는 이유:
  - 연산 속도가 매우 느림 (대칭키보다 100-1000 배)
  - 대용량 데이터 암호화에 부적합

실제 통신 과정:

Step 1: 대칭키 생성 (발신자)
  ┌─────────────────────────────────────────┐
  │  실제 데이터 암호화용 대칭키 생성       │
  │  (AES-256 등)                           │
  └─────────────────────────────────────────┘

Step 2: 대칭키 암호화 (발신자)
  ┌─────────────────────────────────────────┐
  │  대칭키를 수신자의 공개키로 암호화      │
  │  (비대칭키의 역할: 키 교환 문제 해결)   │
  └─────────────────────────────────────────┘

Step 3: 대칭키 복호화 (수신자)
  ┌─────────────────────────────────────────┐
  │  수신자가 개인키로 대칭키 복호화        │
  │  이제 양쪽이 동일한 대칭키 공유         │
  └─────────────────────────────────────────┘

Step 4: 데이터 통신 (양방향)
  ┌─────────────────────────────────────────┐
  │  공유된 대칭키로 대용량 데이터 암호화   │
  │  빠른 속도로 양방향 통신                │
  └─────────────────────────────────────────┘

결론:
  비대칭키 = 대칭키를 안전하게 전달하기 위한 '금고'
  대칭키 = 실제 데이터 암호화용 '열쇠'
```

                    │   Root CA       │
                    │  (최상위 인증기관) │

## Root CA (루트 인증기관)

### 개념

```
┌─────────────────────────────────────────────────────────────┐
│              Root CA 란?                                    │
└─────────────────────────────────────────────────────────────┘

Root CA (루트 인증기관):
  - 인증서 계층 구조의 최상위
  - 자기 자신의 인증서에 자신이 서명 (자기 서명)
  - 전 세계적으로 신뢰받는 기관만 운영
  - OS 와 브라우저에 미리 내장됨

주요 Root CA:
  - DigiCert
  - Sectigo (구 Comodo CA)
  - Let's Encrypt
  - GlobalSign
  - GoDaddy
  - Amazon Trust Services
```

### Root CA 의 특징

```
┌─────────────────────────────────────────────────────────────┐
│              Root CA 의 특징                                │
└─────────────────────────────────────────────────────────────┘

1. 자기 서명 (Self-Signed)
   - Root CA 인증서는 자신이 자신에게 서명
   - 상위 인증기관 없음
   - "나는 나를 신뢰한다" 선언

2. 미리 내장됨
   - Windows, macOS, Linux 에 기본 내장
   - Chrome, Firefox, Edge 에 기본 내장
   - 수백 개의 Root CA 등록됨

3. 엄격한 관리
   - 오프라인 보관 (공격 차단)
   - HSM(Hardware Security Module) 사용
   - 접근 로그 철저한 기록
   - 물리적 보안 (금고, 생체인식)

4. 긴 유효기간
   - 보통 10-25 년
   - 자주 변경하면 모든 OS/브라우저 업데이트 필요
```

### Root CA 인증서 예시

```bash
# Root CA 인증서 확인 (예: DigiCert)
openssl x509 -in digicert-root.crt -text -noout

# 출력 예시:
Certificate:
    Data:
        Version: 3 (0x2)
        Serial Number: 01:db:91:6b:...
        Signature Algorithm: sha256WithRSAEncryption
    Issuer: C=US, O=DigiCert Inc, OU=www.digicert.com, 
            CN=DigiCert Global Root CA
              ↑
              └─ Issuer 와 Subject 가 동일 (자기 서명)
    Validity:
        Not Before: Nov 10 00:00:00 2006 GMT
        Not After:  Nov 10 00:00:00 2031 GMT
              ↑
              └─ 25 년 유효기간
    Subject: C=US, O=DigiCert Inc, OU=www.digicert.com, 
             CN=DigiCert Global Root CA
              ↑
              └─ Issuer 와 동일
```

### Root CA 신뢰 저장소

```bash
# Linux (RedHat/CentOS) 에서 Root CA 목록
ls -la /etc/pki/ca-trust/source/anchors/

# Linux (Ubuntu/Debian) 에서 Root CA 목록
ls -la /usr/share/ca-certificates/mozilla/

# macOS 에서 Root CA 목록
security find-certificate -a -p /System/Library/Keychains/SystemRootCertificates.keychain

# 확인된 Root CA 수: 약 100-200 개
```

## Intermediate CA (중간 인증기관)

### 개념

```
┌─────────────────────────────────────────────────────────────┐
│              Intermediate CA 란?                            │
└─────────────────────────────────────────────────────────────┘

Intermediate CA (중간 인증기관):
  - Root CA 와 최종 사용자 사이의 중간 계층
  - Root CA 로부터 인증을 받음
  - 실제 인증서 발급 업무 수행
  - 여러 계층 가능 (Intermediate → Intermediate)

왜 필요한가?
  1. Root CA 보호 (온라인 노출 방지)
  2. 확장성 (여러 중간 CA 로 업무 분산)
  3. 폐기 용이 (중간 CA 만 폐기, Root 는 유지)
```

### 계층 구조 예시

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 계층 구조 예시                          │
└─────────────────────────────────────────────────────────────┘

Level 0: DigiCert Global Root CA
           │
           │ 서명
           ▼
Level 1: DigiCert SHA2 Secure Server CA (Intermediate)
           │
           │ 서명
           ▼
Level 2: DigiCert TLS RSA SHA256 2020 CA1 (Intermediate)
           │
           │ 서명
           ▼
Level 3: *.google.com (최종 인증서)

체인 검증:
  1. *.google.com 인증서를 DigiCert TLS RSA SHA256 2020 CA1 로 검증
  2. DigiCert TLS RSA SHA256 2020 CA1 을 DigiCert SHA2 Secure Server CA 로 검증
  3. DigiCert SHA2 Secure Server CA 를 DigiCert Global Root CA 로 검증
  4. DigiCert Global Root CA 는 OS/브라우저에 내장됨 (신뢰함)
  5. 모든 체인이 연결되면 신뢰함!
```

### Intermediate CA 의 역할

```
┌─────────────────────────────────────────────────────────────┐
│              Intermediate CA 의 역할                        │
└─────────────────────────────────────────────────────────────┘

1. 실제 인증서 발급
   - 웹사이트 인증서
   - 코드 서명 인증서
   - 이메일 인증서
   - Kubernetes 인증서

2. Root CA 보호
   - Intermediate CA 는 온라인 (발급용)
   - Root CA 는 오프라인 (보관용)
   - Intermediate CA 해킹당해도 Root CA 는 안전
   - Intermediate CA 만 폐기하고 재발급

3. 정책 시행
   - 검증 정책 (도메인 소유 확인 등)
   - 인증서 수명 관리
   - 폐기 목록 (CRL) 관리
```

## 인증서 발급 과정

### 단계별 발급 과정

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 발급 과정                               │
└─────────────────────────────────────────────────────────────┘

1. 키 쌍 생성
   ┌─────────────────┐
   │  개인키 생성    │
   │  (RSA 2048+)    │
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │  공개키 추출    │
   └────────┬────────┘
            │
            ▼
2. CSR (Certificate Signing Request) 생성
   ┌─────────────────┐
   │  CSR 생성       │
   │  - 공개키 포함  │
   │  - 정보 입력    │
   │    (도메인, 조직)│
   └────────┬────────┘
            │
            ▼
3. CA 에 제출 및 검증
   ┌─────────────────┐
   │  CA 에 CSR 제출 │
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │  도메인 소유    │
   │  확인 (DV)      │
   │  또는           │
   │  조직 확인 (OV) │
   └────────┬────────┘
            │
            ▼
4. 인증서 서명 및 발급
   ┌─────────────────┐
   │  CA 개인키로    │
   │  CSR 서명       │
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │  인증서 발급    │
   │  (.crt 파일)    │
   └─────────────────┘
```

### CSR 생성 예시

```bash
# 1. 개인키 생성
openssl genrsa -out mysite.key 2048

# 2. CSR 생성
openssl req -new -key mysite.key -out mysite.csr \
  -subj "/C=KR/ST=Seoul/L=Seoul/O=MyCompany/CN=mysite.com"

# CSR 내용 확인
openssl req -in mysite.csr -text -noout

# 출력 예시:
Certificate Request:
    Data:
        Version: 1 (0x0)
        Subject: C=KR, ST=Seoul, L=Seoul, O=MyCompany, CN=mysite.com
        Subject Public Key Info:
            Public Key Algorithm: rsaEncryption
                Public-Key: (2048 bit)
                Modulus: 00:a1:b2:c3:...
                Exponent: 65537 (0x10001)
```

### CA 에 제출 (예: Let's Encrypt)

```bash
# Certbot 을 사용한 Let's Encrypt 인증서 발급
certbot certonly --webroot -w /var/www/html -d mysite.com

# 출력:
# - /etc/letsencrypt/live/mysite.com/fullchain.pem (인증서 체인)
# - /etc/letsencrypt/live/mysite.com/privkey.pem (개인키)
```

## 인증서 체인 검증

### 체인 검증 과정

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 체인 검증                               │
└─────────────────────────────────────────────────────────────┘

클라이언트가 서버 인증서를 받을 때:

1. 서버 인증서 확인
   ┌─────────────────────────────┐
   │  Subject: mysite.com        │
   │  Issuer: DigiCert TLS CA    │
   │  서명: (DigiCert TLS CA)    │
   └─────────────────────────────┘

2. 중간 CA 인증서로 검증
   ┌─────────────────────────────┐
   │  Subject: DigiCert TLS CA   │
   │  Issuer: DigiCert Root CA   │
   │  서명: (DigiCert Root CA)   │
   └─────────────────────────────┘

3. Root CA 로 검증
   ┌─────────────────────────────┐
   │  Subject: DigiCert Root CA  │
   │  Issuer: DigiCert Root CA   │
   │  서명: (자기 자신)          │
   └─────────────────────────────┘
   │
   └─ OS/브라우저에 내장됨 (신뢰함)

4. 체인 완성
   mysite.com → DigiCert TLS CA → DigiCert Root CA (신뢰)
   
   모든 서명이 유효하면 → 신뢰함!
```

## CA 의 역할: 신뢰의 중재자

### 왜 CA 가 필요한가?

```
┌─────────────────────────────────────────────────────────────┐
│              CA 가 필요한 이유                              │
└─────────────────────────────────────────────────────────────┘

문제: 공개키를 누구나 배포할 수 있다면?

  해커가 수신자를 사칭해서 가짜 공개키를 뿌릴 수 있음!

시나리오:
  1. 해커가 google.com 사칭
  2. 가짜 공개키 배포
  3. 사용자가 가짜 공개키로 암호화
  4. 해커가 개인키로 복호화 → 모든 정보 유출
  5. 해커가 진짜 서버에 전달 (중간자 공격)

해결책: 신뢰할 수 있는 제 3 자 (CA) 가 보증!
```

### 인증서의 구성: "누가 발행했고, 누구의 키인가?"

```
┌─────────────────────────────────────────────────────────────┐
│              디지털 인증서의 구성                           │
└─────────────────────────────────────────────────────────────┘

인증서 = 디지털 신분증

┌─────────────────────────────────────────────────────────────┐
│            디지털 인증서 (예시)                             │
├─────────────────────────────────────────────────────────────┤
│  발급 대상 (Subject):                                       │
│    - 웹사이트 주소 (도메인): www.google.com                 │
│    - 소유자: Google LLC                                     │
│    - 조직: Google                                           │
│                                                             │
│  공개키 (Public Key):                                       │
│    - 알고리즘: RSA 2048 bit                                 │
│    - 값: 00:a1:b2:c3:... (실제 공개키)                      │
│                                                             │
│  발급자 (Issuer):                                           │
│    - CA 이름: DigiCert Inc                                  │
│    - CA 계층: DigiCert Global Root CA                       │
│                                                             │
│  디지털 서명:                                               │
│    - 알고리즘: SHA256 with RSA                              │
│    - 값: 30:82:01:22:... (CA 가 개인키로 서명한 값)          │
│                                                             │
│  유효기간:                                                   │
│    - Not Before: Jan  1 00:00:00 2024 GMT                  │
│    - Not After:  Dec 31 23:59:59 2024 GMT                  │
│                                                             │
│  일련번호:                                                  │
│    - 01:ab:cd:ef:... (고유 식별자)                          │
└─────────────────────────────────────────────────────────────┘

핵심:
  CA 가 자신의 개인키로 이 모든 정보를 서명
  → 위조 불가능
  → 브라우저가 CA 공개키로 검증 가능
```

### 신뢰의 사슬 (Chain of Trust)

```
┌─────────────────────────────────────────────────────────────┐
│              신뢰의 사슬                                    │
└─────────────────────────────────────────────────────────────┘

질문: "CA 는 어떻게 믿나요?"

답변: 계층적인 신뢰 구조!

Level 0: Root CA (최상위)
  ┌─────────────────────────────────────────┐
  │  - 전 세계적으로 공인된 최상위 기관     │
  │  - 공개키가 브라우저/OS 에 미리 내장    │
  │  - 예: DigiCert Global Root CA          │
  └─────────────────────────────────────────┘
           │ 서명
           ▼
Level 1: Intermediate CA (중간)
  ┌─────────────────────────────────────────┐
  │  - Root CA 로부터 인증받음              │
  │  - 실제 발급 업무 수행                  │
  │  - 예: DigiCert SHA2 Secure Server CA   │
  └─────────────────────────────────────────┘
           │ 서명
           ▼
Level 2: End Entity (최종 인증서)
  ┌─────────────────────────────────────────┐
  │  - 웹사이트 인증서                      │
  │  - 예: www.google.com 인증서            │
  └─────────────────────────────────────────┘

검증 과정:
  1. 사용자가 웹사이트에 접속 → 서버가 인증서 전송
  2. 브라우저가 내장된 Root CA 공개키로 검증
  3. 인증서의 CA 서명을 복호화
  4. 해커가 위조했다면 서명 불일치 → 경고!
  5. 모든 체인이 연결되면 → 신뢰!
```

## 브라우저는 Root CA 를 어떻게 신뢰하는가?

### 핵심 질문

```
┌─────────────────────────────────────────────────────────────┐
│              신뢰의 시작점은 어디인가?                      │
└─────────────────────────────────────────────────────────────┘

질문: 
  "브라우저에 저장된 루트 CA 의 공개키로 인증서를 검증한다는데,
   그 브라우저의 루트 CA 는 어떻게 신뢰할 수 있는가?"

이는 "신뢰의 사슬의 끝은 어디인가?"라는 근본적인 질문입니다.

답변: 
  '신뢰의 강제 주입'과 '엄격한 관리 체계' 때문입니다.
```

### 1. 루트 인증서의 '사전 탑재' (Hard-coded Trust)

```
┌─────────────────────────────────────────────────────────────┐
│              루트 인증서 사전 탑재                          │
└─────────────────────────────────────────────────────────────┘

누가 결정하나:
  ┌─────────────────────────────────────────┐
  │  대형 플랫폼 제조사가 결정              │
  │  - 마이크로소프트 (Windows)             │
  │  - 애플 (macOS, iOS)                    │
  │  - 구글 (Android, Chrome)               │
  │  - 모질라 (Firefox)                     │
  └─────────────────────────────────────────┘

어떻게 저장되나:
  ┌─────────────────────────────────────────┐
  │  '루트 인증서 프로그램' 운영            │
  │  - 엄격한 심사 기준                   │
  │  - 정기적인 감사                       │
  │  - 문제 발생 시 즉시 퇴출              │
  └─────────────────────────────────────────┘

저장 위치:
  ┌─────────────────────────────────────────┐
  │  '신뢰할 수 있는 루트 인증 기관 저장소'│
  │  - Windows: Certificate Manager         │
  │  - macOS: Keychain Access               │
  │  - Linux: /etc/ssl/certs/               │
  │  - Chrome: OS 저장소 사용               │
  │  - Firefox: 자체 저장소 (cert9.db)      │
  └─────────────────────────────────────────┘

저장된 Root CA 수:
  - Windows: 약 100 개
  - macOS: 약 150 개
  - Firefox: 약 200 개

우리가 소프트웨어를 설치하는 행위 자체가
  "이 제조사가 보증하는 CA 들을 믿겠다"는
  암묵적인 동의가 됩니다.
```

### 2. 엄격한 감사와 'WebTrust' 인증

```
┌─────────────────────────────────────────────────────────────┐
│              Root CA 인증 기준                              │
└─────────────────────────────────────────────────────────────┘

아무 기업이나 루트 CA 가 될 수 없음!

심사 기준:

1. 물리적 보안
   ┌─────────────────────────────────────────┐
   │  - 지진, 화재 대비 시설                 │
   │  - 군사적 수준의 침입 방어              │
   │  - 생체인식 접근 통제                   │
   │  - 24 시간 CCTV 모니터링                │
   └─────────────────────────────────────────┘

2. 운영 감사
   ┌─────────────────────────────────────────┐
   │  - WebTrust 또는 ETSI 국제 표준 감사   │
   │  - 제 3 의 전문 회계/보안 법인 수행     │
   │  - 매년 정기 감사                       │
   │  - 불시 감사 가능                       │
   └─────────────────────────────────────────┘

3. 키 관리
   ┌─────────────────────────────────────────┐
   │  - HSM(Hardware Security Module) 사용  │
   │  - 다중 서명 (여러 사람이 동시에 승인) │
   │  - 오프라인 보관 (공격 차단)            │
   │  - 접근 로그 철저한 기록                │
   └─────────────────────────────────────────┘

4. 인증서 발급 정책
   ┌─────────────────────────────────────────┐
   │  - 도메인 소유 엄격한 확인             │
   │  - 조직 실사 (OV, EV 인증서)            │
   │  - 부적절한 발급 금지                   │
   │  - 폐기 목록 (CRL) 관리                 │
   └─────────────────────────────────────────┘
```

### 3. '신뢰의 파기' (Root Store Distrust)

```
┌─────────────────────────────────────────────────────────────┐
│              Root CA 퇴출 사례                              │
└─────────────────────────────────────────────────────────────┘

만약 특정 루트 CA 가 해킹당하거나 보안 절차를 어기면?

조치:
  ┌─────────────────────────────────────────┐
  │  플랫폼 제조사가 해당 CA 를             │
  │  '블랙리스트'에 올리고                  │
  │  브라우저 업데이트를 통해               │
  │  그 CA 의 공개키를 삭제                 │
  └─────────────────────────────────────────┘

결과:
  ┌─────────────────────────────────────────┐
  │  해당 CA 가 발급한 전 세계 모든         │
  │  웹사이트는 한순간에                    │
  │  "안전하지 않음" 경고가 뜸              │
  │  → 사실상 폐업 수준의 타격             │
  └─────────────────────────────────────────┘

실제 사례:

1. DigiNotar (2011 년)
   ┌─────────────────────────────────────────┐
   │  - 네덜란드 CA                          │
   │  - 해킹당하여 500 개 이상 위조 인증서   │
   │    발급 (google.com, facebook.com 등)   │
   │  - 모든 브라우저에서 퇴출               │
   │  - 파산                                   │
   └─────────────────────────────────────────┘

2. WoSign / StartCom (2016 년)
   ┌─────────────────────────────────────────┐
   │  - 중국 CA                              │
   │  - 부적절한 인증서 발급 적발            │
   │  - GitHub, Mozilla, Google 등에서 퇴출 │
   │  - 신뢰 상실                              │
   └─────────────────────────────────────────┘

3. Symantec (2017 년)
   ┌─────────────────────────────────────────┐
   │  - 부적절한 인증서 발급 관행 적발       │
   │  - Google 이 32,000 개 인증서 신뢰 중단 │
   │  - CA 사업부 DigiCert 에 매각            │
   └─────────────────────────────────────────┘
```

### 4. 신뢰의 시작점 (Trust Anchor)

```
┌─────────────────────────────────────────────────────────────┐
│              신뢰의 시작점은 어디인가?                      │
└─────────────────────────────────────────────────────────────┘

궁극적인 답변:

우리가 루트 CA 를 믿는 이유는
  → 우리가 사용하는 컴퓨터와 브라우저 제조사를 믿기 때문입니다.

제조사:
  - 구글 (Chrome, Android)
  - 마이크로소프트 (Windows, Edge)
  - 애플 (Safari, macOS, iOS)
  - 모질라 (Firefox)

이들이:
  "우리가 꼼꼼히 검사해 보니 이 CA 들은 믿어도 돼"
  라고 보증하며 소프트웨어 안에 미리 넣어둔 것입니다.

신뢰의 흐름:

  사용자
    │
    │ 신뢰함
    ▼
  브라우저/OS 제조사 (구글, MS, 애플 등)
    │
    │ 신뢰할 수 있는 CA 만 선별
    ▼
  Root CA (DigiCert, Sectigo, Let's Encrypt 등)
    │
    │ 인증서 발급 및 보증
    ▼
  웹사이트 (www.google.com, www.naver.com 등)

결국:
  신뢰의 시작점 (Trust Anchor) 은
  하드웨어와 소프트웨어를 만드는
  거대 플랫폼 기업들의 관리 능력에 기반하고 있습니다.
```

## 요약: 신뢰가 형성되는 흐름

```
┌─────────────────────────────────────────────────────────────┐
│              신뢰 형성 흐름 요약                            │
└─────────────────────────────────────────────────────────────┘

1. 수신자가 "내 공개키 여기 있어!"라고 외치면
   → 믿기 힘들음 (해커일 수 있음)

2. 하지만 "공신력 있는 CA 가 도장을 찍어준
   내 신분증 (인증서) 여기 있어!"라고 하면
   → 브라우저가 확인 시작

3. 브라우저는 미리 가지고 있던 도장
   (루트 CA 공개키) 로 확인
   → CA 서명 검증

4. 모든 체인이 연결되면
   → 안심하고 통신 시작

핵심:
  비대칭키의 신뢰 문제는
  "이미 내 컴퓨터에 저장된
   신뢰할 수 있는 기관 목록" 덕분에
  해결되는 셈입니다.
```

## Kubernetes 와 CA

### Kubernetes 의 CA 구조

```
┌─────────────────────────────────────────────────────────────┐
│              Kubernetes 의 CA 구조                          │
└─────────────────────────────────────────────────────────────┘

kubeadm 으로 클러스터 생성 시:

/etc/kubernetes/pki/
├── ca.crt              # Kubernetes Root CA
├── ca.key              # Kubernetes Root CA 개인키
├── front-proxy-ca.crt  # Front Proxy CA
├── front-proxy-ca.key  # Front Proxy CA 개인키
├── apiserver.crt       # API Server 서버 인증서
├── apiserver.key       # API Server 개인키
├── apiserver-kubelet-client.crt  # API Server → kubelet
├── apiserver-kubelet-client.key
├── controller-manager.crt
├── controller-manager.key
├── scheduler.crt
├── scheduler.key
└── etcd/
    ├── ca.crt          # etcd 전용 CA
    ├── ca.key
    ├── server.crt      # etcd 서버 인증서
    ├── server.key
    ├── peer.crt        # etcd 피어 인증서
    └── peer.key
```

### Kubernetes CA 특징

```
┌─────────────────────────────────────────────────────────────┐
│              Kubernetes CA 특징                             │
└─────────────────────────────────────────────────────────────┘

1. 자체 CA 운영
   - DigiCert 같은 외부 CA 사용 안 함
   - 클러스터 내부 자체 CA
   - kubeadm 이 자동 생성

2. 여러 CA 사용
   - ca.crt: 기본 CA (Pod, ServiceAccount)
   - front-proxy-ca.crt: 애그리게이션 레이어
   - etcd/ca.crt: etcd 전용

3. 유효기간
   - 기본 1 년
   - kubeadm certs check-expiration 로 확인
   - 만료 전 갱신 필요
```

### Kubernetes 인증서 확인

```bash
# CA 인증서 확인
openssl x509 -in /etc/kubernetes/pki/ca.crt -text -noout

# 출력 예시:
Certificate:
    Data:
        Version: 3 (0x2)
        Serial Number: 01:ab:cd:...
        Signature Algorithm: sha256WithRSAEncryption
    Issuer: CN = kubernetes  # ← 자기 서명 (Root CA)
    Validity:
        Not Before: Jan  1 00:00:00 2024 GMT
        Not After:  Jan  1 00:00:00 2025 GMT  # 1 년
    Subject: CN = kubernetes  # ← Issuer 와 동일

# API Server 인증서 확인
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -text -noout

# 출력 예시:
Certificate:
    Issuer: CN = kubernetes  # ← ca.crt 가 서명
    Subject: CN = kube-apiserver
    Validity:
        Not Before: Jan  1 00:00:00 2024 GMT
        Not After:  Jan  1 00:00:00 2025 GMT
```

### Kubernetes 인증서 만료 확인

```bash
# kubeadm 클러스터에서 만료일 확인
kubeadm certs check-expiration

# 출력 예시:
[certs] Certificate "apiserver" will expire in 364d23h59m
[certs] Certificate "apiserver-kubelet-client" will expire in 364d23h59m
[certs] Certificate "front-proxy-client" will expire in 364d23h59m
[certs] Certificate "etcd-server" will expire in 364d23h59m
[certs] Certificate "etcd-peer" will expire in 364d23h59m
[certs] Certificate "etcd-healthcheck-client" will expire in 364d23h59m
[certs] Certificate "apiserver-etcd-client" will expire in 364d23h59m

# 만료 임박 경고:
[certs] WARNING: Certificate "apiserver" expires soon!
```

### 인증서 갱신

```bash
# 모든 인증서 갱신
kubeadm certs renew all

# 개별 인증서 갱신
kubeadm certs renew apiserver
kubeadm certs renew apiserver-kubelet-client
kubeadm certs renew controller-manager
kubeadm certs renew scheduler
kubeadm certs renew etcd-server
kubeadm certs renew etcd-peer
kubeadm certs renew front-proxy-client

# 갱신 후 Pod 재시작
kubectl -n kube-system delete pod -l component=kube-apiserver
kubectl -n kube-system delete pod -l component=kube-controller-manager
kubectl -n kube-system delete pod -l component=kube-scheduler
kubectl -n kube-system delete pod -l component=etcd
```

## 공개 CA vs 자체 CA

### 비교

```
┌─────────────────────────────────────────────────────────────┐
│          공개 CA vs 자체 CA                                 │
├─────────────────────────────────────────────────────────────┤
│  특성          │  공개 CA          │  자체 CA              │
├────────────────┼───────────────────┼───────────────────────┤
│  예시          │  DigiCert,        │  kubeadm ca.crt       │
│                │  Let's Encrypt    │                       │
│  비용          │  유료/무료        │  무료                 │
│  신뢰 범위     │  인터넷 전체      │  클러스터 내부        │
│  발급 시간     │  수분~수일        │  즉시                 │
│  유효기간      │  90 일~2 년        │  원하는 대로          │
│  용도          │  공개 웹사이트    │  내부 통신            │
└─────────────────────────────────────────────────────────────┘
```

### 사용 사례

```
┌─────────────────────────────────────────────────────────────┐
│              CA 선택 가이드                                 │
└─────────────────────────────────────────────────────────────┘

공개 CA 사용 (DigiCert, Let's Encrypt):
  ✓ 외부 사용자에게 공개된 서비스
  ✓ 웹 브라우저에서 접속
  ✓ 신뢰 체인 필요
  ✓ 예: 쇼핑몰, 은행, 공공 웹사이트

자체 CA 사용 (kubeadm ca.crt):
  ✓ 클러스터 내부 통신
  ✓ 신뢰 체인 불필요 (직접 CA 배포)
  ✓ 빠른 발급/갱신
  ✓ 예: 
    - API Server ↔ kubelet
    - API Server ↔ controller-manager
    - etcd 피어 간 통신
    - kubectl 사용자 인증서
```

## 요약

```
┌─────────────────────────────────────────────────────────────┐
│                    핵심 요약                                │
├─────────────────────────────────────────────────────────────┤
│  1. Root CA (루트 인증기관)                                 │
│     - 계층 구조 최상위                                      │
│     - 자기 서명 (Self-Signed)                               │
│     - OS/브라우저에 미리 내장                               │
│     - 엄격한 물리적 보안 (오프라인 보관)                   │
│     - 긴 유효기간 (10-25 년)                                 │
│                                                             │
│  2. Intermediate CA (중간 인증기관)                         │
│     - Root CA 와 최종 사용자 사이                           │
│     - Root CA 로부터 인증받음                               │
│     - 실제 인증서 발급 업무                                 │
│     - Root CA 보호 (해킹 시 폐기 용이)                      │
│                                                             │
│  3. 인증서 체인 검증                                        │
│     - 최종 인증서 → 중간 CA → Root CA                       │
│     - 모든 서명 검증                                        │
│     - Root CA 가 신뢰되면 전체 체인 신뢰                    │
│                                                             │
│  4. Kubernetes CA                                           │
│     - 자체 CA 운영 (kubeadm 이 자동 생성)                   │
│     - 여러 CA 사용 (기본, front-proxy, etcd)                │
│     - 유효기간 1 년 (정기 갱신 필요)                         │
│     - 클러스터 내부 통신용 (외부 공개 안 함)                │
│                                                             │
│  5. 공개 CA vs 자체 CA                                      │
│     - 공개 웹사이트: 공개 CA (DigiCert, Let's Encrypt)       │
│     - 내부 통신: 자체 CA (kubeadm ca.crt)                   │
└─────────────────────────────────────────────────────────────┘
```

**CA 는 디지털 신뢰의 기반입니다. Root CA 는 인터넷의 신뢰를, Kubernetes CA 는 클러스터 내부 신뢰를 담당합니다.**
