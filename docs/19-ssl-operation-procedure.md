# SSL/TLS 동작 절차

SSL(Secure Sockets Layer) 과 TLS(Transport Layer Security) 의 동작 원리를 상세히 설명합니다.

---

## 핵심 개념: 하이브리드 암호화 방식

```
┌─────────────────────────────────────────────────────────────┐
│        SSL/TLS = 비대칭키 + 대칭키의 완벽한 조합            │
└─────────────────────────────────────────────────────────────┘

핵심 질문:
  "왜 두 가지 암호화 방식을 다 쓸까?"

답변:
  각자의 단점을 보완하기 위해!

┌─────────────────────────────────────────────────────────────┐
│              대칭키 vs 비대칭키                             │
├─────────────────────────────────────────────────────────────┤
│  대칭키 (AES 등)               │  비대칭키 (RSA, ECDSA 등)  │
├────────────────────────────────┼────────────────────────────┤
│  ✓ 빠름 (1000 배 빠름)         │  ✓ 느림 (연산 복잡)        │
│  ✓ 대량 데이터 암호화 적합     │  ✓ 키 교환 문제 해결       │
│  ✗ 키 배포 문제                │  ✓ 인증/서명 가능          │
│  ✗ n(n-1)/2 개 키 필요         │  ✓ n 개 키만 필요          │
└────────────────────────────────┴────────────────────────────┘

해결책:
  "비대칭키로 대칭키를 안전하게 전달하고,
   이후에는 대칭키로 빠르게 통신한다!"
```

### 하이브리드 방식의 동작 원리

```
┌─────────────────────────────────────────────────────────────┐
│              SSL/TLS 하이브리드 암호화                      │
└─────────────────────────────────────────────────────────────┘

         핸드셰이크 단계                  데이터 전송 단계
         (초기 연결)                     (본 통신)
         ─────────────                   ─────────────
              │                                 │
              ▼                                 │
    ┌───────────────────┐                       │
    │   비대칭키 암호화 │                       │
    │   (RSA, ECDHE)    │                       │
    │                   │                       │
    │  • 인증서 교환    │                       │
    │  • 신원 확인      │                       │
    │  • Premaster      │                       │
    │    Secret 암호화  │                       │
    │  • Session Key    │                       │
    │    생성           │                       │
    │                   │                       │
    │  느림 (1 회만)    │                       │
    └─────────┬─────────┘                       │
              │                                 │
              │ Session Key 공유 완료           │
              ▼                                 ▼
    ┌───────────────────────────────────────────────────────┐
    │         대칭키 암호화 (AES-256 등)                    │
    │                                                       │
    │  • HTTP 요청/응답 암호화                              │
    │  • 대량 데이터 전송                                   │
    │  • 빠른 속도                                          │
    │                                                       │
    │  빠름 (지속적)                                        │
    └───────────────────────────────────────────────────────┘

비유:
  비대칭키 = 귀중품 (대칭키) 을 운반하는 장갑차
  대칭키 = 일상적인 화물 수송 트럭

  처음만 장갑차 사용 (1 회, 느려도 괜찮음)
  이후엔 트럭으로 빠르게 수송 (지속적, 빠름)
```

```
  │     - 안전함 (개인키만 알 수 있음)      │
  │                                         │
  │  2. 대칭키로 데이터 통신 (지속적)       │
  │     - 빠름 (1000 배)                    │
  │     - 실용적                             │
  │                                         │
  │  결과: 안전하고 빠른 통신!              │
  └─────────────────────────────────────────┘
```

## SSL 과 TLS 의 관계

```
┌─────────────────────────────────────────────────────────────┐
│              SSL vs TLS                                     │
└─────────────────────────────────────────────────────────────┘

SSL (Secure Sockets Layer):
  - Netscape 가 1994 년 개발
  - SSL 1.0 (공개 안 됨)
  - SSL 2.0 (1995 년, 취약점 많음)
  - SSL 3.0 (1996 년, 마지막 SSL 버전)

TLS (Transport Layer Security):
  - IETF 가 1999 년 표준화 (SSL 3.0 기반)
  - TLS 1.0 (1999 년, SSL 3.0 과 호환)
  - TLS 1.1 (2006 년)
  - TLS 1.2 (2008 년, 현재 널리 사용)
  - TLS 1.3 (2018 년, 최신, 성능/보안 강화)

현재:
  - SSL 2.0/3.0: 사용 금지 (취약)
  - TLS 1.0/1.1: 사용 금지 (2020 년부터)
  - TLS 1.2: 현재 표준
  - TLS 1.3: 최신 표준 (권장)

비고:
  - 여전히 "SSL"이라고 부르지만 실제로는 TLS 사용
  - "SSL 인증서" → 실제로는 TLS 인증서
  - "SSL 포트 443" → 실제로는 TLS 포트
```

```

│              TLS 프로토콜 스택                              │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│         Application Layer               │
│  (HTTP, FTP, SMTP, IMAP, etc.)          │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│         TLS Record Protocol             │
│  - 데이터 분할/조립                     │
│  - 압축 (선택)                          │
│  - 암호화/복호화                        │
│  - 무결성 검증 (MAC)                    │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│         TLS Handshake Protocol          │
│  - 인증서 교환                          │
│  - 키 협상                              │
│  - 세션 설정                            │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│         TCP (Transmission Control)      │
│  - 신뢰할 수 있는 전송                  │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│         IP (Internet Protocol)          │
│  - 패킷 라우팅                          │
└─────────────────────────────────────────┘
```

## TLS 핸드셰이크 상세 절차

### TLS 1.2 핸드셰이크 (전통적 방식)

```
┌─────────────────────────────────────────────────────────────┐
│              TLS 1.2 핸드셰이크                             │
└─────────────────────────────────────────────────────────────┘

클라이언트                              서버
   │                                      │
   │  1. ClientHello                      │
   │  - TLS 버전 (1.2)                    │
   │  - Client Random (32 바이트)         │
   │  - Session ID                        │
   │  - Cipher Suites (목록)              │
   │  - Compression Methods               │
   │  - Extensions (SNI, ALPN, etc.)      │
   │─────────────────────────────────────▶│
   │                                      │
   │  2. ServerHello                      │
   │  - TLS 버전 (선택)                   │
   │  - Server Random (32 바이트)         │
   │  - Session ID                        │
   │  - Cipher Suite (선택)               │
   │  - Compression Method                │
   │◀─────────────────────────────────────│
   │                                      │
   │  3. Certificate                      │
   │  - 서버 인증서                       │
   │  - CA 체인                           │
   │◀─────────────────────────────────────│
   │                                      │
   │  4. ServerKeyExchange (선택)         │
   │  - 키 교환 추가 정보                 │
   │  - DHE/ECDHE 파라미터                │
   │◀─────────────────────────────────────│
   │                                      │
   │  5. ServerHelloDone                  │
   │  - "내 준비 끝"                      │
   │◀─────────────────────────────────────│
   │                                      │
   │  6. ClientKeyExchange                │
   │  - Premaster Secret                  │
   │  - 서버 공개키로 암호화              │
   │─────────────────────────────────────▶│
   │                                      │
   │  7. ChangeCipherSpec                 │
   │  - "이제부터 암호화!"                │
   │─────────────────────────────────────▶│
   │                                      │
   │  8. Finished (암호화)                │
   │  - 핸드셰이크 해시                   │
   │  - Session Key 로 암호화             │
   │─────────────────────────────────────▶│
   │                                      │
   │  9. ChangeCipherSpec                 │
   │  - "나도 암호화!"                    │
   │◀─────────────────────────────────────│
   │                                      │
   │  10. Finished (암호화)               │
   │  - 핸드셰이크 해시                   │
   │  - Session Key 로 암호화             │
   │◀─────────────────────────────────────│
   │                                      │
   │  ✓ 핸드셰이크 완료                   │
   │  Application Data (암호화 통신)      │
   │◀════════════════════════════════════▶│
   └──────────────────────────────────────┘

총 10 단계, 2 RTT (Round Trip Time)
```

### TLS 1.3 vs 1.2 비교

```
┌─────────────────────────────────────────────────────────────┐
│              TLS 1.3 vs TLS 1.2                             │
└─────────────────────────────────────────────────────────────┘

TLS 1.2:
  - 10 단계 메시지
  - 2 RTT (Round Trip Time)
  - ServerKeyExchange 별도
  - ChangeCipherSpec 사용
  - 더 많은 암호화 스위트

TLS 1.3:
  - 6 단계 메시지
  - 1 RTT (50% 빠름!)
  - Key Share 통합
  - ChangeCipherSpec 제거
  - 안전한 암호화 스위트만
  - 0-RTT 모드 지원 (재접속 시)

성능 비교:
  ┌──────────────┬──────────┬──────────┐
  │  항목        │  TLS 1.2 │  TLS 1.3 │
  ├──────────────┼──────────┼──────────┤
  │ 핸드셰이크   │  2 RTT   │  1 RTT   │
  │ 메시지 수    │  10 개    │  6 개     │
  │ 초기 지연    │  높음    │  낮음    │
  │ 재접속       │  1 RTT   │  0-RTT   │
  │ 보안성       │  높음    │  매우높음│
  └──────────────┴──────────┴──────────┘
```

## TLS Record Protocol

### 레코드 구조

```
┌─────────────────────────────────────────────────────────────┐
│              TLS Record 구조                                │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    TLS Record                               │
├──────────┬──────────┬──────────┬──────────┬─────────────────┤
│ Content  │ Version  │ Length   │ Fragment │   MAC         │
│  Type    │          │          │          │  (선택)       │
│ (1 바이트)│(2 바이트)│(2 바이트)│ (가변)   │  (가변)       │
├──────────┼──────────┼──────────┼──────────┼─────────────────┤
│   0x17   │  0x0303  │   0x0020 │  암호화   │   HMAC        │
│ (App Data)│ (TLS 1.2)│  (32)    │  데이터  │  (선택)       │
└──────────┴──────────┴──────────┴──────────┴─────────────────┘
              ↑
              └─ TLS 버전
                 0x0300 = SSL 3.0
                 0x0301 = TLS 1.0
                 0x0302 = TLS 1.1
                 0x0303 = TLS 1.2
                 0x0304 = TLS 1.3
```

### Content Type

```
┌─────────────────────────────────────────────────────────────┐
│              Content Type                                   │
└─────────────────────────────────────────────────────────────┘

0x00 - ChangeCipherSpec
  - 암호화 모드 전환 알림

0x01 - Alert
  - 경고/오류 메시지
  - 예: handshake_failure, bad_certificate

0x02 - Handshake
  - 핸드셰이크 메시지
  - ClientHello, ServerHello, Certificate, etc.

0x03 - Application
  - 실제 애플리케이션 데이터
  - HTTP 요청/응답 등
```

### 레코드 처리 과정

```
┌─────────────────────────────────────────────────────────────┐
│              TLS Record 처리 과정                           │
└─────────────────────────────────────────────────────────────┘

1. 분할 (Fragmentation)
   ┌─────────────────────────────────────────┐
   │  애플리케이션 데이터 (최대 16KB)        │
   │  ↓                                      │
   │  여러 개의 Fragment 로 분할              │
   │  (각각 최대 16384 바이트)                │
   └─────────────────────────────────────────┘

2. 압축 (Compression, 선택)
   ┌─────────────────────────────────────────┐
   │  Fragment                               │
   │  ↓                                      │
   │  압축 (CRIME 공격으로 TLS 1.3 에서 제거)│
   └─────────────────────────────────────────┘

3. MAC 추가 (Message Authentication Code)
   ┌─────────────────────────────────────────┐
   │  Fragment                               │
   │  ↓                                      │
   │  + HMAC (Hash-based MAC)                │
   │  - 무결성 검증                          │
   │  - 위변조 방지                          │
   └─────────────────────────────────────────┘

4. 암호화 (Encryption)
   ┌─────────────────────────────────────────┐
   │  Fragment + MAC                         │
   │  ↓                                      │
   │  + 암호화 (AES, ChaCha20, etc.)         │
   │  - Session Key 사용                     │
   └─────────────────────────────────────────┘

5. 헤더 추가
   ┌─────────────────────────────────────────┐
   │  + Content Type (1 바이트)              │
   │  + Version (2 바이트)                   │
   │  + Length (2 바이트)                    │
   │                                         │
   │  = 완성된 TLS Record                    │
   └─────────────────────────────────────────┘
```

## 키 유도 과정 (Key Derivation)

### 하이브리드 키 유도 흐름

```
┌─────────────────────────────────────────────────────────────┐
│          하이브리드 키 유도: 비대칭 → 대칭                  │
└─────────────────────────────────────────────────────────────┘

전체 흐름:

1. 비대칭키로 대칭키 보호 (핸드셰이크)
   ┌─────────────────────────────────────────┐
   │  Premaster Secret (48 바이트 난수)      │
   │         ↓                               │
   │  서버 공개키로 암호화 (RSA/ECDHE)       │
   │         ↓                               │
   │  전송 (중간에 탈취해도 안전)            │
   │         ↓                               │
   │  서버 개인키로 복호화                   │
   └─────────────────────────────────────────┘

2. 양쪽 모두 대칭키 생성
   ┌─────────────────────────────────────────┐
   │  입력:                                  │
   │  - Premaster Secret                     │
   │  - Client Random (평문)                 │
   │  - Server Random (평문)                 │
   │         ↓                               │
   │  PRF/HKDF 함수                          │
   │         ↓                               │
   │  출력:                                  │
   │  - Session Key (대칭키, AES-256 등)     │
   │  - MAC Key (무결성 검증)                │
   │  - IV (초기화 벡터)                     │
   └─────────────────────────────────────────┘

3. 대칭키로 통신 (본 통신)
   ┌─────────────────────────────────────────┐
   │  Session Key 로 모든 데이터 암호화      │
   │  - AES-256-GCM                          │
   │  - 1000 배 빠름                         │
   │  - 실시간 통신 가능                     │
   └─────────────────────────────────────────┘

핵심:
  비대칭키 = 대칭키를 보호하는 "금고"
  대칭키 = 실제 데이터를 암호화하는 "열쇠"
```

### TLS 1.2 키 유도

```
┌─────────────────────────────────────────────────────────────┐
│              TLS 1.2 키 유도                                │
└─────────────────────────────────────────────────────────────┘

입력:
  - Client Random (32 바이트)
  - Server Random (32 바이트)
  - Premaster Secret (48 바이트)

Step 1: Master Secret 생성
  ┌─────────────────────────────────────────┐
  │  Master Secret = PRF(                   │
  │    Premaster Secret,                    │
  │    "master secret",                     │
  │    Client Random + Server Random        │
  │  )                                      │
  │                                         │
  │  결과: 48 바이트                        │
  └─────────────────────────────────────────┘

Step 2: Session Keys 생성
  ┌─────────────────────────────────────────┐
  │  Key Block = PRF(                       │
  │    Master Secret,                       │
  │    "key expansion",                     │
  │    Server Random + Client Random        │
  │  )                                      │
  │                                         │
  │  Key Block 분할:                        │
  │  - Client Write MAC Key                 │
  │  - Server Write MAC Key                 │
  │  - Client Write Key (AES 키)            │
  │  - Server Write Key (AES 키)            │
  │  - Client Write IV (초기화 벡터)        │
  │  - Server Write IV (초기화 벡터)        │
  └─────────────────────────────────────────┘

결과:
  - 양방향 암호화 키
  - 양방향 MAC 키
  - 양방향 IV
```

### TLS 1.3 키 유도 (HKDF)

```
┌─────────────────────────────────────────────────────────────┐
│              TLS 1.3 키 유도 (HKDF)                         │
└─────────────────────────────────────────────────────────────┘

TLS 1.3 은 HKDF (HMAC-based Key Derivation Function) 사용:

Step 1: Early Secret
  ┌─────────────────────────────────────────┐
  │  Early Secret = HKDF-Extract(           │
  │    salt = 0,                            │
  │    IKM = 0                              │
  │  )                                      │
  └─────────────────────────────────────────┘

Step 2: Handshake Secret
  ┌─────────────────────────────────────────┐
  │  Handshake Secret = HKDF-Extract(       │
  │    salt = Early Secret,                 │
  │    IKM = Shared Secret (ECDH)           │
  │  )                                      │
  │                                         │
  │  생성:                                  │
  │  - Client Handshake Traffic Secret      │
  │  - Server Handshake Traffic Secret      │
  └─────────────────────────────────────────┘

Step 3: Master Secret
  ┌─────────────────────────────────────────┐
  │  Master Secret = HKDF-Extract(          │
  │    salt = Handshake Secret,             │
  │    IKM = 0                              │
  │  )                                      │
  │                                         │
  │  생성:                                  │
  │  - Client Application Traffic Secret    │
  │  - Server Application Traffic Secret    │
  │  - Exporter Master Secret               │
  │  - Resumption Master Secret             │
  └─────────────────────────────────────────┘

HKDF 장점:
  - 더 안전한 키 유도
  - 순방향 비밀성 보장
  - 키 계층 구조 명확
```

## 암호화 스위트 (Cipher Suite)

### TLS 1.2 암호화 스위트

```
┌─────────────────────────────────────────────────────────────┐
│              TLS 1.2 Cipher Suite                           │
└─────────────────────────────────────────────────────────────┘

구조:
  TLS_키교환_인증_암호화_MAC

예시:
  TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
       │     │      │         │
       │     │      │         └─ MAC (SHA384)
       │     │      └─────────── 암호화 (AES-256-GCM)
       │     └────────────────── 인증 (RSA)
       └──────────────────────── 키교환 (ECDHE)

주요 구성 요소:

1. 키교환 (Key Exchange)
   - RSA: RSA 키 교환 (순방향 비밀성 없음)
   - DHE: Diffie-Hellman Ephemeral (순방향 비밀성 있음)
   - ECDHE: Elliptic Curve DHE (빠름, 안전함)

2. 인증 (Authentication)
   - RSA: RSA 인증서
   - ECDSA: ECDSA 인증서

3. 암호화 (Bulk Encryption)
   - AES_128_CBC: AES-128 CBC 모드
   - AES_256_CBC: AES-256 CBC 모드
   - AES_128_GCM: AES-128 GCM 모드 (권장)
   - AES_256_GCM: AES-256 GCM 모드 (권장)
   - CHACHA20_POLY1305: ChaCha20 (모바일 최적화)

4. MAC (Message Authentication Code)
   - SHA256: 256 비트 해시
   - SHA384: 384 비트 해시
```

### TLS 1.3 암호화 스위트 (단순화)

```
┌─────────────────────────────────────────────────────────────┐
│              TLS 1.3 Cipher Suite                           │
└─────────────────────────────────────────────────────────────┘

TLS 1.3 은 5 개 스위트만 허용 (안전한 것만):

1. TLS_AES_128_GCM_SHA256
2. TLS_AES_256_GCM_SHA384
3. TLS_CHACHA20_POLY1305_SHA256
4. TLS_AES_128_CCM_SHA256
5. TLS_AES_128_CCM_8_SHA256

구조:
  TLS_암호화_MAC

차이점:
  - 키교환 분리 (항상 ECDHE 사용)
  - 인증 분리 (인증서 타입과 무관)
  - AEAD 만 허용 (GCM, CCM, Poly1305)
  - CBC 모드 제거 (취약)
  - RC4, 3DES, SHA1 제거 (취약)
```

## 세션 관리

### 세션 재개 (Session Resumption)

```
┌─────────────────────────────────────────────────────────────┐
│              세션 재개 (Session Resumption)                 │
└─────────────────────────────────────────────────────────────┘

목적:
  - 재접속 시 핸드셰이크 단축
  - 성능 향상

방법 1: Session ID (TLS 1.2)
  ┌─────────────────────────────────────────┐
  │  1. 첫 접속:                            │
  │     - 전체 핸드셰이크                   │
  │     - 서버가 Session ID 발급            │
  │     - 서버에 세션 정보 저장             │
  │                                         │
  │  2. 재접속:                             │
  │     - ClientHello 에 Session ID 포함    │
  │     - 서버가 세션 정보 찾기             │
  │     - 찾으면 짧은 핸드셰이크            │
  │     - 1 RTT                             │
  └─────────────────────────────────────────┘

방법 2: Session Ticket (TLS 1.2/1.3)
  ┌─────────────────────────────────────────┐
  │  1. 첫 접속:                            │
  │     - 전체 핸드셰이크                   │
  │     - 서버가 암호화된 Ticket 발급       │
  │     - 클라이언트가 저장                 │
  │                                         │
  │  2. 재접속:                             │
  │     - ClientHello 에 Ticket 포함        │
  │     - 서버가 Ticket 복호화              │
  │     - 세션 정보 복원                    │
  │     - 짧은 핸드셰이크                   │
  │     - 1 RTT                             │
  └─────────────────────────────────────────┘

방법 3: 0-RTT (TLS 1.3 만)
  ┌─────────────────────────────────────────┐
  │  1. 첫 접속:                            │
  │     - 전체 핸드셰이크                   │
  │     - PSK (Pre-Shared Key) 발급         │
  │                                         │
  │  2. 재접속:                             │
  │     - ClientHello 에 PSK 포함           │
  │     - 즉시 데이터 전송 (0-RTT!)         │
  │     - 가장 빠름                         │
  │     - 주의: 리플레이 공격 가능성        │
  └─────────────────────────────────────────┘
```

## 경고 (Alert) 프로토콜

### Alert 메시지

```
┌─────────────────────────────────────────────────────────────┐
│              TLS Alert 프로토콜                             │
└─────────────────────────────────────────────────────────────┘

구조:
  ┌──────────┬──────────┐
  │  Level   │  Code    │
  │ (1 바이트)│(1 바이트)│
  └──────────┴──────────┘

Level:
  1 - Warning (경고, 계속 가능)
  2 - Fatal (치명적, 연결 종료)

주요 Alert Codes:

10 - close_notify
  - 정상 종료 알림

20 - handshake_failure
  - 핸드셰이크 실패

40 - bad_certificate
  - 인증서 오류

42 - bad_certificate_hash_value
  - 인증서 해시 불일치

46 - unsupported_certificate
  - 지원하지 않는 인증서

47 - certificate_revoked
  - 인증서 폐기됨

48 - certificate_expired
  - 인증서 만료됨

49 - certificate_unknown
  - 알 수 없는 인증서

70 - protocol_version
  - 지원하지 않는 프로토콜 버전

80 - internal_error
  - 내부 오류
```
