# OAuth 2.0 개념 정리

OAuth 2.0 은 권한 부여 프로토콜로, "인증은 사용자가, 권한은 클라이언트가"라는 핵심 개념을 가집니다.

---

## OAuth 2.0 핵심 개념

### 가장 중요한 한 문장

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│    "인증 (Authentication) 은 사용자가 수행하고,             │
│     권한 부여 (Authorization) 는 클라이언트가 받는다"       │
│                                                             │
└─────────────────────────────────────────────────────────────┘

이것이 OAuth 2.0 의 핵심입니다!
```

### 인증 vs 권한 부여

```
┌─────────────────────────────────────────────────────────────┐
│          인증 (Authentication) vs 권한 부여 (Authorization) │
└─────────────────────────────────────────────────────────────┘

인증 (Authentication):
  - 질문: "누구인가?" (Who are you?)
  - 주체: 사용자 (User)
  - 목적: 사용자 신원 확인
  - 수단: 아이디/비밀번호, 지문, 얼굴인식 등
  - 결과: "이 사람은 홍길동이다"
  - 수행자: 사용자가 직접 수행

권한 부여 (Authorization):
  - 질문: "무엇을 할 수 있는가?" (What can you do?)
  - 주체: 클라이언트 (애플리케이션)
  - 목적: 리소스 접근 권한 획득
  - 수단: Access Token
  - 결과: "이 앱은 캘린더를 읽을 수 있다"
  - 수행자: 클라이언트가 위임받음
```

### OAuth 2.0 의 역할 분리

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 역할 분리                                │
└─────────────────────────────────────────────────────────────┘

┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│   사용자    │      │  클라이언트 │      │    서버     │
│   (User)    │      │  (Client)   │      │  (Server)   │
│             │      │             │      │             │
│  인증 수행  │      │  권한 위임  │      │  권한 부여  │
│  (로그인)   │      │  (토큰 요청)│      │  (토큰 발급)│
│             │      │             │      │             │
│  "나는      │      │  "내가      │      │  "확인됨,   │
│   홍길동    │      │  대신       │      │  이 권한    │
│   입니다"   │      │  작업할게요"│      │  드립니다"  │
└─────────────┘      └─────────────┘      └─────────────┘
     ↑                      ↑                      ↑
     │                      │                      │
     └──────────────────────┴──────────────────────┘
                          │
                    OAuth 2.0 프로토콜
```

## OAuth 2.0 시나리오

### 실제 예시: Google 캘린더 앱

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 실제 시나리오                            │
└─────────────────────────────────────────────────────────────┘

상황:
  - 사용자는 "CalendarApp" 이라는 앱을 사용
  - CalendarApp 이 Google 캘린더에 접근해야 함

문제:
  ❌ 잘못된 방식:
     - 사용자가 Google 아이디/비밀번호를 CalendarApp 에 입력
     - CalendarApp 이 비밀번호로 Google 에 로그인
     - 문제: 비밀번호 유출 위험, CalendarApp 이 모든 Google 접근 가능

  ✓ OAuth 2.0 방식:
     - 사용자는 Google 에서만 로그인 (인증)
     - CalendarApp 은 권한만 위임받음 (권한 부여)
     - CalendarApp 은 캘린더만 접근 가능, 이메일/사진은 접근 불가
```

### OAuth 2.0 흐름

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 인증/권한 부여 흐름                      │
└─────────────────────────────────────────────────────────────┘

사용자        CalendarApp         Google
  │                │                  │
  │  1. 앱 실행    │                  │
  │───────────────▶│                  │
  │                │                  │
  │                │  2. 권한 요청    │
  │                │─────────────────▶│
  │                │                  │
  │  3. Google 로그인 페이지          │
  │◀─────────────────────────────────│
  │                │                  │
  │  4. 로그인 (인증)                 │
  │  "나는 홍길동입니다"              │
  │─────────────────────────────────▶│
  │                │                  │
  │  5. 권한 동의 페이지              │
  │  "CalendarApp 이 캘린더에         │
  │   접근하는 것을 허용하시겠습니까?"│
  │◀─────────────────────────────────│
  │                │                  │
  │  6. 동의 (권한 부여)              │
  │  "캘린더 접근만 허용"             │
  │─────────────────────────────────▶│
  │                │                  │
  │                │  7. Access Token │
  │                │  "캘린더 접근권한"│
  │                │◀─────────────────│
  │                │                  │
  │                │  8. 토큰으로     │
  │                │  캘린더 조회     │
  │                │─────────────────▶│
  │                │                  │
  │                │  9. 캘린더 데이터│
  │                │◀─────────────────│
  │                │                  │
  │  10. 캘린더 표시                │
  │◀───────────────│                  │
  └───────────────────────────────────┘

핵심:
  - 단계 4: 사용자가 직접 인증 (로그인)
  - 단계 6: 사용자가 클라이언트에 권한 위임
  - 단계 7: 클라이언트가 권한 (토큰) 획득
  - 단계 8: 클라이언트가 권한으로 리소스 접근
```

## OAuth 2.0 주요 용어

### 4 가지 역할

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 4 가지 역할                              │
└─────────────────────────────────────────────────────────────┘

1. Resource Owner (리소스 소유자)
   - 사용자 (User)
   - 리소스의 주인
   - 인증을 수행하는 주체
   - 권한을 위임하는 주체
   - 예: Google 계정 소유자

2. Client (클라이언트)
   - 애플리케이션 (Application)
   - 리소스에 접근하려는 주체
   - 권한을 위임받는 주체
   - Access Token 을 사용하는 주체
   - 예: CalendarApp

3. Resource Server (리소스 서버)
   - API 서버
   - 보호된 리소스를 보유
   - Access Token 으로 접근 제어
   - 예: Google Calendar API

4. Authorization Server (인증 서버)
   - 인증 및 권한 부여 수행
   - Access Token 발급
   - 사용자의 인증 확인
   - 사용자의 권한 동의 확인
   - 예: Google Accounts
```

### 용어 관계도

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 용어 관계                                │
└─────────────────────────────────────────────────────────────┘

                    ┌─────────────────────────┐
                    │  Authorization Server   │
                    │  (인증 + 권한 부여)     │
                    │  - 사용자 인증          │
                    │  - 토큰 발급            │
                    └───────────┬─────────────┘
                                │
                    ┌───────────┴─────────────┐
                    │                         │
          ┌─────────▼─────────┐     ┌─────────▼─────────┐
          │  Resource Owner   │     │  Resource Server  │
          │  (사용자)         │     │  (API 서버)       │
          │  - 인증 수행      │     │  - 리소스 보유    │
          │  - 권한 위임      │     │  - 토큰 검증      │
          └─────────┬─────────┘     └─────────┬─────────┘
                    │                         │
                    │    권한 위임            │    토큰 제시
                    │                         │
                    └───────────┬─────────────┘
                                │
                    ┌───────────▼─────────────┐
                    │  Client                 │
                    │  (애플리케이션)         │
                    │  - 권한 요청            │
                    │  - 토큰 사용            │
                    └─────────────────────────┘
```

## Access Token

### Access Token 이란?

```
┌─────────────────────────────────────────────────────────────┐
│          Access Token                                       │
└─────────────────────────────────────────────────────────────┘

정의:
  - 권한 부여의 증거
  - 클라이언트가 리소스에 접근할 수 있는 "열쇠"
  - 사용자가 위임한 권한의 범위와 기간을 포함

특징:
  ✓ 문자열 (일반적으로 JWT)
  ✓ 유효기간 있음 (보통 1 시간)
  ✓ 범위 (scope) 포함
  ✓ 재사용 가능 (만료될 때까지)
  ✓ 클라이언트가 보관 및 사용

예시:
  ya29.a0AfH6SMBxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

### Access Token 의 역할

```
┌─────────────────────────────────────────────────────────────┐
│          Access Token 의 역할                               │
└─────────────────────────────────────────────────────────────┘

Access Token 은 다음을 나타냅니다:

1. "이 클라이언트는 사용자의 동의를 받았습니다"
   - 사용자가 직접 권한 위임

2. "이 클라이언트는 이 범위까지만 접근 가능합니다"
   - 예: 캘린더 읽기만, 쓰기 불가

3. "이 클라이언트는 이 기간까지만 접근 가능합니다"
   - 예: 1 시간 동안 유효

4. "이 클라이언트는 이 리소스에만 접근 가능합니다"
   - 예: 캘린더는 가능, 이메일은 불가
```

### Access Token 사용 예시

```bash
# Access Token 으로 API 호출
curl -X GET \
  -H "Authorization: Bearer ya29.a0AfH6SM..." \
  https://www.googleapis.com/calendar/v3/calendars/primary/events

# 응답:
{
  "kind": "calendar#events",
  "items": [
    {
      "summary": "팀 미팅",
      "start": { "dateTime": "2024-01-15T10:00:00" },
      ...
    }
  ]
}

# Access Token 이 없으면:
curl -X GET \
  https://www.googleapis.com/calendar/v3/calendars/primary/events

# 응답:
{
  "error": {
    "code": 401,
    "message": "Request is missing required authentication credential"
  }
}
```

```
│  6. Google 이 CalendarApp 에 Access Token 발급             │
│     "이 토큰으로 캘린더만 접근 가능"                       │
│                                                             │
│  ↑ 이 단계부터가 권한 부여 (Authorization)                 │
│  ↑ 클라이언트가 위임받음                                   │
└─────────────────────────────────────────────────────────────┘
```

## OAuth 2.0 의 핵심 가치

### 1. 비밀번호 공유 금지

```
┌─────────────────────────────────────────────────────────────┐
│          비밀번호 공유 금지                                 │
└─────────────────────────────────────────────────────────────┘

❌ OAuth 2.0 이전:
  사용자 → CalendarApp: "내 Google 비밀번호는 hong1234 입니다"
  CalendarApp → Google: 비밀번호로 로그인
  문제: CalendarApp 이 모든 Google 리소스 접근 가능

✓ OAuth 2.0:
  사용자 → Google: 비밀번호 입력 (CalendarApp 에게 절대 노출 안 함)
  Google → CalendarApp: Access Token 만 발급
  장점: CalendarApp 은 캘린더만 접근 가능, 비밀번호는 안전
```

### 2. 최소 권한 원칙

```
┌─────────────────────────────────────────────────────────────┐
│          최소 권한 원칙 (Least Privilege)                   │
└─────────────────────────────────────────────────────────────┘

사용자가 위임하는 권한은 필요한 최소한으로:

❌ 잘못된 예:
  "CalendarApp 이 내 Google 계정의 모든 것에 접근 허용"
  - 캘린더, 이메일, 사진, 드라이브, 연락처... 모두 접근 가능

✓ 올바른 예:
  "CalendarApp 이 내 캘린더 읽기만 허용"
  - 캘린더 읽기만 가능
  - 이메일, 사진, 드라이브는 접근 불가
  - 쓰기도 불가
```

### 3. 권한 취소 가능

```
┌─────────────────────────────────────────────────────────────┐
│          권한 취소 가능                                     │
└─────────────────────────────────────────────────────────────┘

사용자는 언제든지 권한을 취소할 수 있음:

Google 계정 → 보안 → 서드파티 액세스 → CalendarApp → 액세스 취소

결과:
  - Access Token 즉시 무효화
  - CalendarApp 은 더 이상 캘린더 접근 불가
  - 비밀번호 변경 불필요 (비밀번호는 공유 안 했으므로)
```

## OAuth 2.0 vs OpenID Connect

### 관계 정리

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 vs OpenID Connect                        │
└─────────────────────────────────────────────────────────────┘

OAuth 2.0:
  - 권한 부여 (Authorization) 프로토콜
  - 목적: 클라이언트가 리소스에 접근할 권한 획득
  - 결과: Access Token
  - 답변: "이 앱은 캘린더를 읽을 수 있다"

OpenID Connect:
  - 인증 (Authentication) 프로토콜
  - OAuth 2.0 위에 구축
  - 목적: 사용자 신원 확인
  - 결과: ID Token
  - 답변: "이 사용자는 홍길동 (hong@example.com) 이다"

관계:
  ┌─────────────────────────────────────────┐
  │  OpenID Connect = OAuth 2.0 + 인증      │
  │                                         │
  │  OAuth 2.0 은 권한 부여만               │
  │  OIDC 가 인증 기능 추가                 │
  └─────────────────────────────────────────┘
```

### 사용 사례

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 vs OIDC 사용 사례                        │
└─────────────────────────────────────────────────────────────┘

OAuth 2.0 만 필요한 경우:
  - 앱이 Google 캘린더에 일정 추가
  - 앱이 Dropbox 에 파일 저장
  - 앱이 GitHub 에 코드 푸시
  - 사용자 신원보다 리소스 접근이 목적

OIDC 가 필요한 경우:
  - 앱에서 Google 계정으로 로그인
  - 앱에서 Facebook 계정으로 로그인
  - 앱에서 회사 계정으로 로그인
  - 사용자 신원 확인이 목적
```

```
│     - 질문: "누구인가?"                                    │
│     - 주체: 사용자                                         │
│     - 수단: 아이디/비밀번호, MFA 등                        │
│     - 결과: 사용자 신원 확인                               │
│     - 수행: 사용자가 직접 Google 등에서 로그인             │
│                                                             │
│  3. 권한 부여 (Authorization)                               │
│     - 질문: "무엇을 할 수 있는가?"                         │
│     - 주체: 클라이언트 (앱)                                │
│     - 수단: Access Token                                   │
│     - 결과: 리소스 접근 권한                               │
│     - 수행: 클라이언트가 사용자로부터 위임                 │
│                                                             │
│  4. Access Token                                            │
│     - 권한 부여의 증거                                     │
│     - 클라이언트가 리소스 접근에 사용                      │
│     - 범위 (scope) 와 유효기간 포함                        │
│     - 비밀번호와 달리 제한된 권한만                        │
│                                                             │
│  5. OAuth 2.0 의 가치                                       │
│     - 비밀번호 공유 금지                                   │
│     - 최소 권한 원칙                                       │
│     - 권한 취소 가능                                       │
│                                                             │
│  6. OAuth 2.0 vs OIDC                                       │
│     - OAuth 2.0: 권한 부여만                               │
│     - OIDC: OAuth 2.0 + 인증                               │
└─────────────────────────────────────────────────────────────┘
```

### OAuth 2.0 개념 도식

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 개념 도식                                │
└─────────────────────────────────────────────────────────────┘

                    사용자 (Resource Owner)
                         │
                         │ 1. 인증 수행 (로그인)
                         │    "나는 홍길동입니다"
                         ▼
                    Authorization Server
                         │
                         │ 2. 권한 동의 확인
                         │    "캘린더만 허용"
                         │
         ┌───────────────┴───────────────┐
         │                               │
         │ 3. Access Token 발급          │
         │    "캘린더 접근 권한"          │
         │                               │
         ▼                               │
    Client (애플리케이션)                │
         │                               │
         │ 4. 토큰으로 리소스 접근       │
         │    "권한 있어요"              │
         │                               │
         ▼                               │
    Resource Server (API) ───────────────┘
         │
         │ 5. 토큰 검증
         │    "확인됨, 접근 허용"
         │
         ▼
    리소스 제공 (캘린더 데이터)

핵심:
  - 인증 (1 단계): 사용자가 직접 수행
  - 권한 부여 (2-5 단계): 클라이언트가 위임받음
```

**OAuth 2.0 은 "인증은 사용자가, 권한 부여는 클라이언트가"라는 명확한 역할 분리를 통해 보안을 강화합니다. 사용자는 비밀번호를 공유하지 않고, 클라이언트는 필요한 최소 권한만 위임받습니다.**

## OAuth 2.0 vs OIDC 핵심 비유

### 문열쇠 vs 신분증

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 vs OIDC 핵심 비유                        │
└─────────────────────────────────────────────────────────────┘

OAuth 2.0:
  ┌─────────────────────────────────────────┐
  │         🗝️  문열쇠 (Access Token)       │
  └─────────────────────────────────────────┘
  - 특정 방 (리소스) 에 접근할 수 있는 열쇠
  - "이 사람은 캘린더 방에 들어갈 수 있다"
  - 누구인지는 모름, 그냥 권한만 있음
  - 인가 (Authorization) 전용

OIDC (OpenID Connect):
  ┌─────────────────────────────────────────┐
  │         🪪  신분증 (ID Token)           │
  └─────────────────────────────────────────┘
  - 사용자의 신원을 증명하는 신분증
  - "이 사람은 홍길동 (hong@example.com) 입니다"
  - 이름, 이메일, 사진 등 신원 정보 포함
  - 인증 (Authentication) 전용
```

## OAuth 2.0 의 근본적인 한계

### 1. OAuth 2.0: "인증 기능이 설계상 빠져 있음"

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 의 근본적인 한계                         │
└─────────────────────────────────────────────────────────────┘

OAuth 2.0 명세서 (RFC 6749) 를 보면:
  - 이 프로토콜의 목적은 오로지 '인가 (Authorization)'
  - 인증 (Authentication) 은 고려되지 않음

문제점:
  ❌ "이 사용자가 누구인가 (Who is he?)"에 대한
     표준화된 응답 방법이 없습니다!

과거의 잘못된 사용:
  ┌─────────────────────────────────────────┐
  │  개발자들이 Access Token 으로            │
  │  "인증처럼" 사용하려고 시도             │
  │                                         │
  │  1. Access Token 받음                   │
  │  2. 유저 정보 API 호출                  │
  │  3. 사용자 정보 얻음                    │
  │                                         │
  │  문제:                                  │
  │  - 표준이 아님                          │
  │  - 서비스마다 구현 방식 제각각          │
  │  - 보안 허점 많음                       │
  └─────────────────────────────────────────┘
```

### 왜 OAuth 2.0 에 인증이 없을까?

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 의 설계 목적                             │
└─────────────────────────────────────────────────────────────┘

OAuth 2.0 이 만들어진 목적 (2012 년):
  - "비밀번호 없이 리소스 접근 권한 위임"
  - 인증이 아니라 권한 부여에 집중

예시:
  - CalendarApp 이 내 캘린더에 접근
  - 나는 CalendarApp 에 비밀번호 주고 싶지 않음
  - Access Token 만 주면 됨
  - 누구인지 (인증) 는 중요하지 않음
  - 무엇을 할 수 있는지 (인가) 만 중요

결과:
  - OAuth 2.0 명세서에 인증 관련 내용 없음
  - Access Token 은 권한만 증명, 신원은 증명 안 함
  - "이 토큰의 주인이 누구인가?"에 대한 답 없음
```

## OIDC: OAuth 2.0 + 인증 (Identity Layer)

### 2. OIDC: "OAuth 2.0 + 인증 (Identity Layer)"

```
┌─────────────────────────────────────────────────────────────┐
│          OIDC = OAuth 2.0 + 인증 레이어                     │
└─────────────────────────────────────────────────────────────┘

OIDC (OpenID Connect, 2014 년):
  - OAuth 2.0 을 확장한 프로토콜
  - OAuth 2.0 의 모든 '인가' 기능 그대로 유지
  - 그 위에 '인증' 기능을 얹음

구조:
  ┌─────────────────────────────────────────┐
  │  OIDC = OAuth 2.0 + ID Token (JWT)      │
  └─────────────────────────────────────────┘

범위:
  ✓ 인가 (Authorization): Access Token
  ✓ 인증 (Authentication): ID Token
  ✓ 둘 다 동시에 처리 가능
```

### OIDC 의 핵심 추가: ID Token

```
┌─────────────────────────────────────────────────────────────┐
│          ID Token (신분증)                                  │
└─────────────────────────────────────────────────────────────┘

ID Token 이 추가됨으로써:

OAuth 2.0 만 있을 때:
  ┌─────────────────────────────────────────┐
  │  Access Token 만 있음                   │
  │                                         │
  │  - 이 토큰으로 사진첩 접근 가능         │
  │  - 이 토큰으로 메일 발송 가능           │
  │  - 하지만 "이 사람이 누구인가?" 모름   │
  │  - '행위 (Action)'만 가능               │
  └─────────────────────────────────────────┘

OIDC 추가 후:
  ┌─────────────────────────────────────────┐
  │  Access Token + ID Token 동시 받음      │
  │                                         │
  │  - Access Token: 사진첩 접근 (인가)     │
  │  - ID Token: "이 사람은 홍길동,         │
  │             이메일 hong@test.com"       │
  │             (인증)                      │
  │  - '행위' + '신원 확인' 모두 가능       │
  └─────────────────────────────────────────┘
```

## 결론: OIDC 가 인가와 인증을 모두 포함하는가?

### 명확한 답변

```
┌─────────────────────────────────────────────────────────────┐
│          결론: OIDC 는 인가 + 인증 모두 포함                │
└─────────────────────────────────────────────────────────────┘

질문: "OIDC 가 인가와 인증을 모두 포함하는가?"

답변: "네, 맞습니다!"

이유:
  ┌─────────────────────────────────────────┐
  │  OIDC 는 OAuth 2.0 이라는               │
  │  튼튼한 '권한 관리' 기반 위에           │
  │  '신분증 (ID Token)'이라는 개념을       │
  │  더한 것입니다                          │
  └─────────────────────────────────────────┘

결과:
  ✓ 인가 (Authorization): Access Token 으로 리소스 접근
  ✓ 인증 (Authentication): ID Token 으로 신원 확인
  ✓ 둘 다 한 번의 흐름으로 처리
```

### 현대적인 서비스에서의 의미

```
┌─────────────────────────────────────────────────────────────┐
│          현대적인 서비스에서의 OIDC                         │
└─────────────────────────────────────────────────────────────┘

"소셜 로그인 기능을 만들겠다" = 사실상 OIDC 사용

예시:
  - "Google 로 로그인" 버튼 → OIDC
  - "Facebook 으로 로그인" 버튼 → OIDC
  - "카카오로 로그인" 버튼 → OIDC
  - "네이버로 로그인" 버튼 → OIDC

왜?
  ✓ 인증 (로그인) 필요: ID Token 으로 사용자 신원 확인
  ✓ 인가 (권한) 필요: Access Token 으로 프로필 등 접근
  ✓ 둘 다 필요 → OIDC 가 정답!
```

### 최종 정리

```
┌─────────────────────────────────────────────────────────────┐
│                    최종 정리                                │
└─────────────────────────────────────────────────────────────┘

1. OAuth 2.0 의 한계:
   - 인증 기능이 설계상 빠져 있음 (RFC 6749)
   - "이 사용자가 누구인가?"에 대한 표준 응답 없음
   - Access Token 만으로는 로그인 불가

2. OIDC 의 해결:
   - OAuth 2.0 + ID Token (JWT)
   - 인가 + 인증 모두 처리
   - 표준화된 로그인 프로세스

3. 실무적 차이:
   - OAuth 2.0 만: Access Token 만 받음 (행위만 가능)
   - OIDC: Access Token + ID Token 동시 받음 (로그인 가능)

4. 결론:
   - OIDC 는 인가와 인증을 모두 포함
   - 현대적인 소셜 로그인 = 사실상 OIDC
   - "소셜 로그인 만들겠다" = OIDC 사용한다는 뜻
```

## OIDC 는 인증을 "수행"하는가, "위임"하는가?

### 핵심 관점: OIDC 의 본질

```
┌─────────────────────────────────────────────────────────────┐
│          OIDC 는 인증을 직접 수행하는가?                    │
└─────────────────────────────────────────────────────────────┘

정답: "위임 (Delegation) 입니다"

OIDC 자체는 인증을 직접 '수행'하는 엔진이라기보다:
  - 외부의 인증 결과를 안전하게 가져와서
  - 내 서비스에 태우는 '규격 (Protocol)'
  - 에 가깝습니다

구조적 이해:
  ┌─────────────────────────────────────────┐
  │  "인증은 외부에 맡기고 (Delegation),    │
  │   그 결과물 (Identity) 을               │
  │   내 서비스에 통합하는                  │
  │   표준화된 통로"                        │
  └─────────────────────────────────────────┘
```

### 그럼 왜 "OIDC 는 인증을 포함한다"고 하는가?

```
┌─────────────────────────────────────────────────────────────┐
│          "인증 포함"이라고 표현하는 3 가지 이유             │
└─────────────────────────────────────────────────────────────┘

질문:
  "위임인데 왜 '포함'이라고 하나요?"

답변:
  실무적인 이유 3 가지가 있습니다!
```

## 이유 1: "무엇을 (What)"에 대한 약속

### OAuth 2.0 vs OIDC 의 약속

```
┌─────────────────────────────────────────────────────────────┐
│          이유 1: "무엇을 (What)"에 대한 약속                │
└─────────────────────────────────────────────────────────────┘

OAuth 2.0:
  ┌─────────────────────────────────────────┐
  │  "여기 열쇠 줄게,                      │
  │   나머지는 네가 알아서 해."             │
  │                                         │
  │  - "열쇠 (Token)"를 주는 방법만 정의    │
  │  - 그 안에 "누구의 정보" 를 넣을지      │
  │    정하지 않음                          │
  │  - 외적 행위 중심                       │
  └─────────────────────────────────────────┘

OIDC:
  ┌─────────────────────────────────────────┐
  │  "여기 열쇠랑 신분증도 같이 줄게.      │
  │   이 신분증 양식은 전 세계 공통이야."   │
  │                                         │
  │  - ID Token 이라는 표준 규격            │
  │  - 사용자 신원 정보 (이름, 이메일 등)   │
  │    반드시 포함하도록 강제               │
  │  - 내적 신원 중심                       │
  └─────────────────────────────────────────┘

핵심:
  OIDC 는 "무엇을 (What)"에 대한 약속이 있음!
  - sub (사용자 고유 ID)
  - email (이메일)
  - name (이름)
  - picture (사진)
  - 등 표준 클레임 강제
```

## 이유 2: "인증 대행" 자체가 현대 보안의 핵심

## 왜 최신 소셜 로그인에 OIDC 가 사용되는가?

### 소셜 로그인 요구사항

```
┌─────────────────────────────────────────────────────────────┐
│          소셜 로그인 요구사항                               │
└─────────────────────────────────────────────────────────────┘

"Google 로 로그인" 버튼을 누르면 필요한 것:

1. 인증 (Authentication)
   - "이 사람이 진짜 Google 계정 소유자인가?"
   - 사용자 신원 확인 (이메일, 이름 등)
   - 필요: 🪪 신분증 (ID Token)

2. 인가 (Authorization)
   - "이 앱이 사용자 프로필에 접근할 권한이 있는가?"
   - 리소스 접근 권한 (프로필, 이메일 등)
   - 필요: 🗝️ 문열쇠 (Access Token)

결론:
  둘 다 필요함! → OIDC 가 정답!
```

### OIDC 의 장점

```
┌─────────────────────────────────────────────────────────────┐
│          OIDC 가 소셜 로그인에 사용되는 이유                │
└─────────────────────────────────────────────────────────────┘

1. 한 번에 두 마리 토끼
   ✓ 인증 (ID Token 으로 사용자 신원 확인)
   ✓ 인가 (Access Token 으로 리소스 접근)
   ✓ 별도 프로세스 불필요, 한 번의 흐름으로 완료

2. OAuth 2.0 인프라 재사용
   ✓ 기존 OAuth 2.0 인증 흐름을 그대로 사용
   ✓ 새로운 프로토콜 아님, 확장일 뿐
   ✓ 기존 라이브러리/인프라 활용 가능

3. 표준화된 클레임
   ✓ email, name, picture 등 표준 필드
   ✓ Google, Facebook, Microsoft 모두 동일 방식
   ✓ 통합 쉬움

4. 보안성
   ✓ ID Token 은 JWT + 서명
   ✓ 위조 불가
   ✓ IdP 가 보증하는 신원
```

## OIDC 는 OAuth 2.0 의 인증 흐름을 재사용하고 확장

### 흐름 비교

```
┌─────────────────────────────────────────────────────────────┐
│          OAuth 2.0 vs OIDC 인증 흐름                        │
└─────────────────────────────────────────────────────────────┘

OAuth 2.0 Authorization Code Flow:
┌─────────────────────────────────────────────────────────────┐
│  1. 클라이언트 → IdP: 인증 요청                            │
│  2. 사용자 → IdP: 로그인 (인증)                            │
│  3. 사용자 → IdP: 권한 동의                                │
│  4. IdP → 클라이언트: 인증 코드                            │
│  5. 클라이언트 → IdP: 토큰 요청                            │
│  6. IdP → 클라이언트: Access Token                         │
│                                                             │
│  결과: 🗝️ Access Token 만 받음 (인가만 가능)              │
└─────────────────────────────────────────────────────────────┘

OIDC Authorization Code Flow:
┌─────────────────────────────────────────────────────────────┐
│  1. 클라이언트 → IdP: 인증 요청 (+scope=openid)            │
│  2. 사용자 → IdP: 로그인 (인증)                            │
│  3. 사용자 → IdP: 권한 동의                                │
│  4. IdP → 클라이언트: 인증 코드                            │
│  5. 클라이언트 → IdP: 토큰 요청                            │
│  6. IdP → 클라이언트: Access Token + ID Token ⭐           │
│                                                             │
│  결과: 🗝️ Access Token + 🪪 ID Token (인가 + 인증)        │
└─────────────────────────────────────────────────────────────┘

차이점:
  - 요청에 scope=openid 추가만!
  - 응답에 ID Token 추가됨
  - 나머지는 완전히 동일!
```

### OIDC 의 확장 요소

```
┌─────────────────────────────────────────────────────────────┐
│          OIDC 가 OAuth 2.0 에 추가한 것                     │
└─────────────────────────────────────────────────────────────┘

1. scope=openid
   - 이 scope 가 있으면 OIDC 요청
   - 없으면 일반 OAuth 2.0 요청

2. ID Token
   - JWT 형식의 토큰
   - 사용자 신원 정보 포함 (sub, email, name 등)
   - IdP 가 서명 (위조 불가)

3. UserInfo Endpoint
   - 추가 사용자 정보 조회 엔드포인트
   - Access Token 으로 보호

4. 표준 클레임
   - sub, email, name, picture 등
   - 모든 IdP 가 동일한 필드 사용

5. Discovery Document
   - /.well-known/openid-configuration
   - IdP 설정 자동 발견

핵심:
  OAuth 2.0 흐름을 그대로 재사용!
  거기에 ID Token 등 인증 요소만 추가!
```

```
OAuth 2.0:
  - 권한 부여 프로토콜
  - 🗝️ 문열쇠 (Access Token) 제공
  - "이 리소스에 접근할 권한 있음"
  - 인가 (Authorization) 전용

OIDC:
  - 인증 프로토콜 (OAuth 2.0 기반)
  - 🗝️ 문열쇠 + 🪪 신분증 (Access + ID Token) 제공
  - "홍길동입니다 + 이 리소스에 접근할 권한 있음"
  - 인증 (Authentication) + 인가 (Authorization)

왜 OIDC 가 소셜 로그인에 사용되는가?
  ✓ 한 번의 흐름으로 인증 + 인가 모두 해결
  ✓ OAuth 2.0 인프라 재사용 (확장일 뿐)
  ✓ 표준화된 사용자 정보 (email, name 등)
  ✓ 보안성 (JWT + 서명)

핵심 문장:
  "OAuth 2.0 이 문열쇠를 제공한다면,
   OIDC 는 신분증을 제공합니다.
   따라서 OIDC 를 사용하면 OAuth 2.0 의 인가 기능과
   인증 기능을 동시에 얻을 수 있어
   최신 소셜 로그인에 OIDC 가 사용됩니다.
   OIDC 는 OAuth 2.0 의 인증 흐름을 재사용하고 확장하여
   ID Token 이라는 신분증 기능을 추가했습니다."
```
