# 인증서를 통한 신뢰할 수 있는 통신 절차

인증서가 어떻게 안전한 통신을 보장하는지 단계별로 설명합니다.

---

## TLS/SSL 통신의 전체 흐름

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 기반 신뢰 통신 전체 흐름                │
└─────────────────────────────────────────────────────────────┘

클라이언트 (브라우저)                    서버 (웹사이트)
       │                                      │
       │  1. ClientHello                      │
       │  (지원 TLS 버전, 암호화 스위트)       │
       │─────────────────────────────────────▶│
       │                                      │
       │  2. ServerHello + 인증서             │
       │  (서버 인증서, CA 체인)              │
       │◀─────────────────────────────────────│
       │                                      │
       │  3. 인증서 검증 (신뢰 확인)          │
       │     - CA 서명 검증                   │
       │     - 유효기간 확인                  │
       │     - 폐기 여부 확인 (CRL/OCSP)      │
       │     - 도메인 일치 확인               │
       │                                      │
       │  4. 키 교환 (대칭키 공유)            │
       │  - Premaster Secret 생성             │
       │  - 서버 공개키로 암호화 전송         │
       │─────────────────────────────────────▶│
       │                                      │
       │  5. 서버가 개인키로 복호화           │
       │     양쪽 모두 대칭키 생성            │
       │                                      │
       │  6. Finished                         │
       │  (암호화 통신 시작)                  │
       │◀════════════════════════════════════▶│
       │                                      │
       │  7. 대칭키 암호화 통신               │
       │  (빠른 속도로 데이터 전송)           │
       │◀════════════════════════════════════▶│
       └──────────────────────────────────────┘
```

---

## 단계별 상세 절차

### 1 단계: ClientHello

```
┌─────────────────────────────────────────────────────────────┐
│  ClientHello (클라이언트 → 서버)                            │
└─────────────────────────────────────────────────────────────┘

클라이언트가 서버에 보내는 정보:

1. 지원 TLS 버전
   - TLS 1.2, TLS 1.3 등

2. 지원 암호화 스위트 목록
   - TLS_AES_256_GCM_SHA384
   - TLS_CHACHA20_POLY1305_SHA256
   - TLS_AES_128_GCM_SHA256

3. 클라이언트 랜덤 (Client Random)
   - 32 바이트 난수
   - 키 생성에 사용

4. 지원 확장
   - SNI (Server Name Indication)
   - ALPN (Application-Layer Protocol Negotiation)
   - Supported Groups (타원곡선)

예시:
  ClientHello:
    TLS Version: TLS 1.3
    Cipher Suites: [TLS_AES_256_GCM, TLS_CHACHA20, ...]
    Client Random: a1b2c3d4e5f6...
    SNI: www.example.com
```

---

### 2 단계: ServerHello + 인증서

```
┌─────────────────────────────────────────────────────────────┐
│  ServerHello + Certificate (서버 → 클라이언트)              │
└─────────────────────────────────────────────────────────────┘

서버가 클라이언트에 보내는 정보:

1. ServerHello
   - 선택된 TLS 버전
   - 선택된 암호화 스위트
   - 서버 랜덤 (Server Random)

2. Certificate (서버 인증서)
   ┌─────────────────────────────────────────┐
   │  서버 인증서                            │
   │  - Subject: CN=www.example.com          │
   │  - Issuer: CN=DigiCert CA               │
   │  - Public Key: RSA 2048 bit             │
   │  - Validity: 2024-01-01 ~ 2025-01-01   │
   │  - SAN: www.example.com, example.com    │
   │  - Signature: (CA 서명)                 │
   └─────────────────────────────────────────┘

3. Certificate Chain (중간 CA 인증서)
   ┌─────────────────────────────────────────┐
   │  중간 CA 인증서                         │
   │  - Subject: CN=DigiCert CA              │
   │  - Issuer: CN=DigiCert Root CA          │
   └─────────────────────────────────────────┘

4. Server Key Exchange (선택)
   - 키 교환을 위한 추가 정보

예시:
  ServerHello:
    TLS Version: TLS 1.3
    Cipher Suite: TLS_AES_256_GCM_SHA384
    Server Random: f6e5d4c3b2a1...
  
  Certificate:
    Version: 3
    Subject: CN=www.example.com
    Issuer: CN=DigiCert Global Root CA
    Public Key: RSA 2048 bit
    Signature: sha256WithRSAEncryption
```

---

### 3 단계: 인증서 검증 (신뢰 확인)

```
┌─────────────────────────────────────────────────────────────┐
│  인증서 검증 (클라이언트 내부 처리)                         │
└─────────────────────────────────────────────────────────────┘

브라우저가 수행하는 5 가지 검증:

1. CA 서명 검증
   ┌─────────────────────────────────────────┐
   │  1. 중간 CA 인증서에서 공개키 추출     │
   │  2. 서버 인증서의 서명을 복호화        │
   │  3. 서버 인증서의 해시값 계산          │
   │  4. 두 값 비교 (일치하면 OK)           │
   │                                         │
   │  5. 중간 CA 인증서를 Root CA 로 검증   │
   │     (Root CA 는 브라우저에 내장됨)      │
   └─────────────────────────────────────────┘
   
   검증 결과:
     ✓ CA 서명 유효함
     ✓ 신뢰할 수 있는 CA 가 발급

2. 유효기간 확인
   ┌─────────────────────────────────────────┐
   │  Not Before: 2024-01-01 00:00:00 UTC   │
   │  Not After:  2025-01-01 00:00:00 UTC   │
   │                                         │
   │  현재 시간: 2024-06-15                  │
   │  → 유효기간 내 ✓                        │
   └─────────────────────────────────────────┘

3. 폐기 여부 확인 (CRL/OCSP)
   ┌─────────────────────────────────────────┐
   │  CRL 확인:                              │
   │  - http://crl3.digicert.com/...         │
   │  - 폐기목록 다운로드 및 확인            │
   │                                         │
   │  OCSP 확인 (더 빠름):                   │
   │  - http://ocsp.digicert.com             │
   │  - 실시간 상태 조회                     │
   │  - 응답: "good" (정상)                  │
   └─────────────────────────────────────────┘
   
   결과: ✓ 폐기되지 않음

4. 도메인 일치 확인
   ┌─────────────────────────────────────────┐
   │  접속한 도메인: www.example.com         │
   │                                         │
   │  인증서 CN: www.example.com ✓           │
   │  인증서 SAN:                              │
   │    - www.example.com ✓                  │
   │    - example.com                        │
   │    - mail.example.com                   │
   └─────────────────────────────────────────┘
   
   결과: ✓ 도메인 일치함

5. 키 용도 확인
   ┌─────────────────────────────────────────┐
   │  Key Usage:                             │
   │    - Digital Signature ✓                │
   │    - Key Encipherment ✓                 │
   │                                         │
   │  Extended Key Usage:                    │
   │    - TLS Web Server Authentication ✓    │
   └─────────────────────────────────────────┘
   
   결과: ✓ TLS 서버 용도로 허용됨

모든 검증 통과 → 신뢰할 수 있는 인증서!
```

### 검증 실패 시나리오

```
┌─────────────────────────────────────────────────────────────┐
│  인증서 검증 실패 사례                                      │
└─────────────────────────────────────────────────────────────┘

1. CA 서명 불일치
   ❌ 오류: "이 사이트의 보안 인증서를 발급한 기관을 신뢰할 수 없습니다"
   원인: 자체 서명 인증서, 알 수 없는 CA

2. 유효기간 만료
   ❌ 오류: "이 사이트의 보안 인증서가 만료되었습니다"
   원인: 인증서 갱신 안 함

3. 도메인 불일치
   ❌ 오류: "이 사이트의 보안 인증서가 요청된 도메인 이름과 일치하지 않습니다"
   원인: 다른 도메인용 인증서 사용

4. 폐기된 인증서
   ❌ 오류: "이 사이트의 보안 인증서가 폐기되었습니다"
   원인: 개인키 유출 등으로 폐기

5. 루트 CA 없음
   ❌ 오류: "이 사이트의 보안 인증서를 확인할 수 없습니다"
   원인: 중간 CA 인증서 누락
```

### 4 단계: 키 교환 (대칭키 공유)

```
┌─────────────────────────────────────────────────────────────┐
│  키 교환 (클라이언트 → 서버)                                │
└─────────────────────────────────────────────────────────────┘

TLS 1.3 기준:

1. 클라이언트가 Premaster Secret 생성
   ┌─────────────────────────────────────────┐
   │  Premaster Secret: 4a3b2c1d...         │
   │  (48 바이트 난수)                       │
   └─────────────────────────────────────────┘

2. 서버 공개키로 암호화
   ┌─────────────────────────────────────────┐
   │  Encrypted = RSA_Encrypt(              │
   │    서버_공개키,                         │
   │    Premaster_Secret                     │
   │  )                                      │
   │                                         │
   │  결과: a1b2c3d4e5f6... (암호문)        │
   └─────────────────────────────────────────┘

3. Client Key Exchange 로 전송
   ┌─────────────────────────────────────────┐
   │  Client Key Exchange:                   │
   │    Encrypted Premaster Secret:          │
   │    a1b2c3d4e5f6...                      │
   └─────────────────────────────────────────┘

4. 서버가 개인키로 복호화
   ┌─────────────────────────────────────────┐
   │  Premaster Secret = RSA_Decrypt(       │
   │    서버_개인키,                         │
   │    Encrypted_Premaster                  │
   │  )                                      │
   │                                         │
   │  결과: 4a3b2c1d... (원본)              │
   └─────────────────────────────────────────┘

5. 양쪽 모두 Session Key 생성
   ┌─────────────────────────────────────────┐
   │  Session Key = KDF(                     │
   │    Client Random,                       │
   │    Server Random,                       │
   │    Premaster Secret                     │
   │  )                                      │
   │                                         │
   │  결과: AES-256 키 (32 바이트)           │
   └─────────────────────────────────────────┘

핵심:
  - Premaster Secret 은 서버 공개키로 암호화됨
  - 중간에 탈취해도 서버 개인키 없이는 복호화 불가
  - 양쪽 모두 동일한 Session Key 생성
```

---

### 5 단계: Finished (암호화 통신 시작)

```
┌─────────────────────────────────────────────────────────────┐
│  Finished (양방향)                                          │
└─────────────────────────────────────────────────────────────┘

1. 클라이언트 → 서버: Finished
   ┌─────────────────────────────────────────┐
   │  Finished:                              │
   │    - 모든 이전 메시지 해시값            │
   │    - Session Key 로 암호화              │
   │    - "내 준비 완료!"                    │
   └─────────────────────────────────────────┘

2. 서버 → 클라이언트: Finished
   ┌─────────────────────────────────────────┐
   │  Finished:                              │
   │    - 모든 이전 메시지 해시값            │
   │    - Session Key 로 암호화              │
   │    - "나도 준비 완료!"                  │
   └─────────────────────────────────────────┘

3. Handshake 완료
   ┌─────────────────────────────────────────┐
   │  ✓ 인증서 검증 완료                     │
   │  ✓ 키 교환 완료                         │
   │  ✓ Session Key 공유 완료                │
   │  ✓ 암호화 통신 준비 완료                │
   └─────────────────────────────────────────┘
```

### 6 단계: 대칭키 암호화 통신

```
┌─────────────────────────────────────────────────────────────┐
│  Application Data (양방향 암호화 통신)                      │
└─────────────────────────────────────────────────────────────┘

이제 모든 데이터는 Session Key 로 암호화됨:

클라이언트 → 서버:
  ┌─────────────────────────────────────────┐
  │  평문: "GET /index.html HTTP/1.1..."    │
  │         ↓ (AES-256 암호화)              │
  │  암호문: 8f3a2b1c9d4e...                │
  │         ↓ 전송                          │
  └─────────────────────────────────────────┘

서버 → 클라이언트:
  ┌─────────────────────────────────────────┐
  │  평문: "HTTP/1.1 200 OK..."             │
  │         ↓ (AES-256 암호화)              │
  │  암호문: 5e6f7a8b9c0d...                │
  │         ↓ 전송                          │
  └─────────────────────────────────────────┘

특징:
  - 대칭키 (AES-256) 사용 → 매우 빠름
  - 양방향 암호화
  - 무결성 검증 (HMAC)
  - 중간자 공격 방지
```

## 신뢰 체인 검증 상세

### 인증서 체인 검증 과정

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 체인 검증 상세                          │
└─────────────────────────────────────────────────────────────┘

전체 체인:
  Root CA (DigiCert Global Root CA)
       │ 서명
       ▼
  Intermediate CA (DigiCert SHA2 Secure Server CA)
       │ 서명
       ▼
  Server Certificate (www.example.com)

검증 단계:

Step 1: 서버 인증서 → 중간 CA 검증
  ┌─────────────────────────────────────────┐
  │  1. 중간 CA 인증서에서 공개키 추출      │
  │     Public Key: RSA 2048 bit            │
  │                                         │
  │  2. 서버 인증서의 서명 복호화           │
  │     Signature → Hash1                   │
  │                                         │
  │  3. 서버 인증서 해시 계산               │
  │     Certificate → Hash2                 │
  │                                         │
  │  4. Hash1 == Hash2 비교                 │
  │     ✓ 일치함 → 서명 유효                │
  └─────────────────────────────────────────┘

Step 2: 중간 CA → Root CA 검증
  ┌─────────────────────────────────────────┐
  │  1. Root CA 인증서에서 공개키 추출      │
  │     (브라우저에 내장됨)                 │
  │                                         │
  │  2. 중간 CA 인증서의 서명 복호화        │
  │     Signature → Hash1                   │
  │                                         │
  │  3. 중간 CA 인증서 해시 계산            │
  │     Certificate → Hash2                 │
  │                                         │
  │  4. Hash1 == Hash2 비교                 │
  │     ✓ 일치함 → 서명 유효                │
  └─────────────────────────────────────────┘

Step 3: Root CA 신뢰 확인
  ┌─────────────────────────────────────────┐
  │  Root CA 가 브라우저 신뢰 저장소에     │
  │  있는지 확인                            │
  │                                         │
  │  ✓ 있음 → 신뢰함                       │
  └─────────────────────────────────────────┘

결과:
  모든 체인 검증 통과 → 서버 인증서 신뢰함!
```

## 보안 속성

### 인증서 기반 통신이 제공하는 보안

```
┌─────────────────────────────────────────────────────────────┐
│              TLS 통신의 보안 속성                           │
└─────────────────────────────────────────────────────────────┘

1. 기밀성 (Confidentiality)
   ┌─────────────────────────────────────────┐
   │  - 대칭키 암호화 (AES-256)             │
   │  - 중간자가 데이터 읽기 불가           │
   │  - 예: 비밀번호, 카드 정보 보호         │
   └─────────────────────────────────────────┘

2. 무결성 (Integrity)
   ┌─────────────────────────────────────────┐
   │  - HMAC 으로 데이터 위변조 검출         │
   │  - 전송 중 변경 감지                    │
   │  - 예: 요청 데이터 조작 방지            │
   └─────────────────────────────────────────┘

3. 인증 (Authentication)
   ┌─────────────────────────────────────────┐
   │  - 서버 신원 확인 (인증서)             │
   │  - 피싱 사이트 방지                    │
   │  - 예: 진짜 은행 사이트 확인            │
   └─────────────────────────────────────────┘

4. 부인 방지 (Non-repudiation)
   ┌─────────────────────────────────────────┐
   │  - 디지털 서명으로 행위 증명           │
   │  - 나중에 부인 불가                     │
   │  - 예: 거래 내역 증명                   │
   └─────────────────────────────────────────┘
```

   ┌─────────────────────────────────────────┐

   │  ❌  https://www.example.com            │
   │                                         │
   │  빨간색 경고                           │
   │  "잠재적 보안 위험"                    │

```
┌─────────────────────────────────────────────────────────────┐
│                    핵심 요약                                │
├─────────────────────────────────────────────────────────────┤
│  1. TLS 핸드셰이크 (6 단계)                                 │

│     - 키 교환 → Finished → 암호화 통신                     │
│                                                             │
│  2. 인증서 검증 (5 가지)                                    │
│     - CA 서명 검증 (체인 검증)                             │
│     - 유효기간 확인                                         │
│     - 폐기 여부 확인 (CRL/OCSP)                            │
│     - 도메인 일치 확인                                     │
│     - 키 용도 확인                                         │
│                                                             │
│  3. 키 교환                                                 │
│     - Premaster Secret 을 서버 공개키로 암호화             │
│     - 서버만 복호화 가능 (개인키)                          │
│     - 양쪽 모두 Session Key 생성                           │
│                                                             │
│  4. 암호화 통신                                             │
│     - 대칭키 (AES-256) 로 빠른 암호화                      │
│     - 기밀성, 무결성, 인증, 부인 방지 제공                 │
│                                                             │
│  5. 신뢰 체인                                               │
│     - 서버 → 중간 CA → Root CA                             │
│     - Root CA 는 브라우저에 내장됨                         │
│     - 모든 서명 검증 통과 시 신뢰                          │
│                                                             │
│  6. 브라우저 표시                                           │
│     - 정상: 🔒 녹색 자물쇠                                 │
│     - 경고: ⚠️ 노란색 경고                                  │
│     - 위험: ❌ 빨간색 경고                                  │
└─────────────────────────────────────────────────────────────┘
```

**인증서 기반 TLS 통신은 5 단계 검증과 키 교환을 통해 신뢰할 수 있는 보안 통신을 보장합니다.**
