# PKI 와 X.509 인증서

공개키 기반구조 (PKI) 와 X.509 인증서에 대해 알아봅니다.

---

## PKI (Public Key Infrastructure) 란?

### 개념

```
┌─────────────────────────────────────────────────────────────┐
│              PKI (공개키 기반구조)                          │
└─────────────────────────────────────────────────────────────┘

PKI (Public Key Infrastructure):
  - 공개키 암호화 기술을 안전하게 사용하기 위한 인프라
  - 인증서, CA, 등록기관, 저장소 등으로 구성
  - "누구의 공개키인가?"를 보증하는 시스템

핵심 질문:
  "이 공개키가 정말로 www.google.com 의 것인가?"
  "해커가 만든 가짜 공개키가 아닌가?"

해결책:
  "신뢰할 수 있는 CA 가 보증한다!"
```

### PKI 의 구성 요소

```
┌─────────────────────────────────────────────────────────────┐
│              PKI 의 구성 요소                               │
└─────────────────────────────────────────────────────────────┘

1. CA (Certification Authority, 인증기관)
   ┌─────────────────────────────────────────┐
   │  - 인증서 발급 및 서명                  │
   │  - Root CA, Intermediate CA             │
   │  - 예: DigiCert, Let's Encrypt          │
   └─────────────────────────────────────────┘

2. RA (Registration Authority, 등록기관)
   ┌─────────────────────────────────────────┐
   │  - 인증서 신청자 신원 확인              │
   │  - 도메인 소유 검증                     │
   │  - CA 를 대신하여 심사                  │
   └─────────────────────────────────────────┘

3. 인증서 저장소 (Certificate Repository)
   ┌─────────────────────────────────────────┐
   │  - 발급된 인증서 저장 및 공개           │
   │  - LDAP 디렉토리 등                     │
   │  - 누구나 열람 가능                     │
   └─────────────────────────────────────────┘

4. CRL (Certificate Revocation List, 폐기목록)
   ┌─────────────────────────────────────────┐
   │  - 만료 전 폐기된 인증서 목록           │
   │  - CA 가 정기적으로 발행                │
   │  - 개인키 유출 시 등록                  │
   └─────────────────────────────────────────┘

5. OCSP (Online Certificate Status Protocol)
   ┌─────────────────────────────────────────┐
   │  - 인증서 상태를 실시간으로 조회        │
   │  - CRL 보다 빠른 응답                   │
   │  - "이 인증서 유효한가요?" 조회         │
   └─────────────────────────────────────────┘

6. 최종 사용자 (End Entity)
   ┌─────────────────────────────────────────┐
   │  - 인증서 소유자 (웹사이트, 사용자)     │
   │  - 키 쌍 생성 및 인증서 신청            │
   │  - 개인키 안전한 보관                   │
   └─────────────────────────────────────────┘
```

### PKI 동작 흐름

```
┌─────────────────────────────────────────────────────────────┐
│              PKI 동작 흐름                                  │
└─────────────────────────────────────────────────────────────┘

Step 1: 키 쌍 생성 (사용자)
  ┌─────────────────────────────────────────┐
  │  - 공개키 + 개인키 생성                 │
  │  - 개인키는 안전하게 보관               │
  └─────────────────────────────────────────┘
           │
           ▼
Step 2: 인증서 신청 (사용자 → RA/CA)
  ┌─────────────────────────────────────────┐
  │  - CSR (Certificate Signing Request)    │
  │    생성                                 │
  │  - CSR 에 공개키 포함                   │
  │  - 신원 정보 입력 (도메인, 조직)        │
  └─────────────────────────────────────────┘
           │
           ▼
Step 3: 신원 검증 (RA)
  ┌─────────────────────────────────────────┐
  │  - 도메인 소유 확인 (DV)                │
  │  - 조직 실사 (OV)                       │
  │  - 강화된 검증 (EV)                     │
  └─────────────────────────────────────────┘
           │
           ▼
Step 4: 인증서 발급 (CA)
  ┌─────────────────────────────────────────┐
  │  - CA 가 CSR 정보 확인                  │
  │  - CA 개인키로 서명                     │
  │  - X.509 인증서 생성                    │
  └─────────────────────────────────────────┘
           │
           ▼
Step 5: 인증서 배포 (사용자)
  ┌─────────────────────────────────────────┐
  │  - 웹 서버에 인증서 설치                │
  │  - HTTPS 통신에 사용                    │
  └─────────────────────────────────────────┘
           │
           ▼
Step 6: 검증 (클라이언트)
  ┌─────────────────────────────────────────┐
  │  - 브라우저가 인증서 수신               │
  │  - CA 공개키로 서명 검증                │
  │  - CRL/OCSP 에서 폐기 여부 확인         │
  │  - 유효기간 확인                         │
  └─────────────────────────────────────────┘
```

## X.509 인증서

### 개념

```
┌─────────────────────────────────────────────────────────────┐
│              X.509 인증서란?                                │
└─────────────────────────────────────────────────────────────┘

X.509:
  - ITU-T (국제전기통신연합) 표준
  - 디지털 인증서 포맷 정의
  - PKI 에서 가장 널리 사용되는 인증서 형식
  - 버전 3 (v3) 이 현재 표준

용도:
  - 웹사이트 (HTTPS/SSL/TLS)
  - 이메일 서명 (S/MIME)
  - 코드 서명
  - 사용자 인증
  - Kubernetes 인증서
```

### X.509 인증서 구조

```
┌─────────────────────────────────────────────────────────────┐
│              X.509 인증서 구조 (v3)                         │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    X.509 인증서                             │
├─────────────────────────────────────────────────────────────┤
│  1. 버전 (Version)                                          │
│     - v1, v2, v3 (현재 표준)                                │
│     - v3 는 확장 필드 지원                                  │
│                                                             │
│  2. 일련번호 (Serial Number)                                │
│     - CA 가 부여하는 고유 식별자                            │
│     - 01:ab:cd:ef:12:34:...                                 │
│                                                             │
│  3. 서명 알고리즘 (Signature Algorithm)                     │
│     - sha256WithRSAEncryption                               │
│     - ecdsa-with-SHA384                                     │
│                                                             │
│  4. 발급자 (Issuer)                                         │
│     - CA 의 식별 정보 (Distinguished Name)                  │
│     - CN=DigiCert Global Root CA, O=DigiCert Inc, C=US      │
│                                                             │
│  5. 유효기간 (Validity)                                      │
│     - Not Before: 2024-01-01 00:00:00 UTC                   │
│     - Not After:  2025-01-01 00:00:00 UTC                   │
│                                                             │
│  6. 소유자 (Subject)                                        │
│     - 인증서 소유자의 식별 정보                             │
│     - CN=www.google.com, O=Google LLC, C=US                 │
│                                                             │
│  7. 공개키 정보 (Subject Public Key Info)                   │
│     - 알고리즘: RSA, ECDSA 등                               │
│     - 공개키 값: 00:a1:b2:c3:...                            │
│                                                             │
│  8. 발급자 고유 식별자 (Issuer Unique Identifier) [선택]    │
│                                                             │
│  9. 소유자 고유 식별자 (Subject Unique Identifier) [선택]   │
│                                                             │
│  10. 확장 (Extensions) [v3 만]                              │
│      - Subject Alternative Name (SAN)                       │
│      - Key Usage                                            │
│      - Extended Key Usage                                   │
│      - CRL Distribution Points                              │
│      - Authority Information Access                         │
│                                                             │
│  11. CA 서명 (CA Signature)                                 │
│      - CA 가 위 모든 정보를 개인키로 서명한 값              │
│      - 30:82:01:22:ab:cd:...                                │
└─────────────────────────────────────────────────────────────┘
```

### Distinguished Name (DN)

```
┌─────────────────────────────────────────────────────────────┐
│              Distinguished Name (식별명)                    │
└─────────────────────────────────────────────────────────────┘

X.509 인증서에서Issuer 와 Subject 를 식별하는 고유 이름:

주요 필드:

  CN (Common Name):
    - 가장 중요한 필드
    - 웹사이트: 도메인 이름 (www.google.com)
    - CA: CA 이름 (DigiCert Global Root CA)
    - 사용자: 사용자 이름 (admin@company.com)

  O (Organization):
    - 조직 이름
    - 예: Google LLC, Microsoft Corporation

  OU (Organizational Unit):
    - 조직 내 부서
    - 예: IT Department, Security Team

  L (Locality):
    - 도시/지역
    - 예: Seoul, San Francisco

  ST (State or Province):
    - 도/주/광역시
    - 예: California, Seoul

  C (Country):
    - 국가 코드 (2 글자)
    - 예: US, KR, GB

  Email:
    - 이메일 주소
    - 예: admin@example.com

예시:
  Issuer: C=US, O=DigiCert Inc, CN=DigiCert Global Root CA
  
  Subject: C=US, ST=California, L=Mountain View,
           O=Google LLC, CN=www.google.com
```

### X.509 v3 확장 (Extensions)

```
┌─────────────────────────────────────────────────────────────┐
│              X.509 v3 확장 필드                             │
└─────────────────────────────────────────────────────────────┘

1. Subject Alternative Name (SAN)
   ┌─────────────────────────────────────────┐
   │  - 하나의 인증서로 여러 도메인 보호     │
   │  - 예: google.com, www.google.com,      │
   │        mail.google.com, drive.google.com│
   │  - 현대 브라우저는 SAN 필수 요구        │
   └─────────────────────────────────────────┘

2. Key Usage
   ┌─────────────────────────────────────────┐
   │  - 이 키로 무엇을 할 수 있는지 정의     │
   │  - digitalSignature: 서명용             │
   │  - keyEncipherment: 키 암호화용         │
   │  - dataEncipherment: 데이터 암호화용    │
   │  - keyAgreement: 키 합의용              │
   │  - keyCertSign: 인증서 서명용 (CA 만)   │
   │  - cRLSign: CRL 서명용 (CA 만)          │
   └─────────────────────────────────────────┘

3. Extended Key Usage (EKU)
   ┌─────────────────────────────────────────┐
   │  - 용도 제한                            │
   │  - serverAuth: 서버 인증 (웹사이트)     │
   │  - clientAuth: 클라이언트 인증          │
   │  - codeSigning: 코드 서명               │
   │  - emailProtection: 이메일 보호         │
   │  - timeStamping: 타임스탬프             │
   └─────────────────────────────────────────┘

4. Basic Constraints
   ┌─────────────────────────────────────────┐
   │  - 이 인증서가 CA 인지 여부             │
   │  - CA:TRUE → CA 인증서                  │
   │  - CA:FALSE → 최종 사용자 인증서        │
   │  - pathlen: 최대 계층 깊이              │
   └─────────────────────────────────────────┘

5. CRL Distribution Points
   ┌─────────────────────────────────────────┐
   │  - CRL (폐기목록) 위치                  │
   │  - http://crl3.digicert.com/...         │
   │  - 브라우저가 자동 확인                 │
   └─────────────────────────────────────────┘

6. Authority Information Access (AIA)
   ┌─────────────────────────────────────────┐
   │  - CA 인증서 위치                       │
   │  - OCSP 서버 위치                       │
   │  - http://ocsp.digicert.com             │
   └─────────────────────────────────────────┘

7. Subject Key Identifier
   ┌─────────────────────────────────────────┐
   │  - 이 인증서의 공개키 해시값            │
   │  - 인증서 식별용                        │
   └─────────────────────────────────────────┘

8. Authority Key Identifier
   ┌─────────────────────────────────────────┐
   │  - 이 인증서를 서명한 CA 의 공개키 해시 │
   │  - 체인 검증용                          │
   └─────────────────────────────────────────┘
```

### X.509 인증서 예시

```bash
# 인증서 내용 확인
openssl x509 -in certificate.crt -text -noout

# 출력 예시:

    Data:
        Version: 3 (0x2)                    ← v3
        Serial Number:
            01:ab:cd:ef:12:34:56:78         ← 일련번호

        Not Before: Jan  1 00:00:00 2024 GMT  ← 유효기간 시작
        Not After:  Dec 31 23:59:59 2024 GMT  ← 유효기간 종료
    Subject: C=US, ST=California, L=Mountain View,
             O=Google LLC, CN=www.google.com  ← 소유자
    Subject Public Key Info:
        Public Key Algorithm: rsaEncryption
            Public-Key: (2048 bit)
            Modulus: 00:a1:b2:c3:...
            Exponent: 65537 (0x10001)
    X509v3 extensions:
        X509v3 Subject Alternative Name:
            DNS:www.google.com, DNS:google.com, DNS:mail.google.com

            Digital Signature, Key Encipherment
        X509v3 Extended Key Usage:
            TLS Web Server Authentication

            CA:FALSE

            Full Name:
              URI:http://crl3.digicert.com/...
        Authority Information Access:
            OCSP - URI:http://ocsp.digicert.com
            CA Issuers - URI:http://cacerts.digicert.com/...
        X509v3 Subject Key Identifier:
            AB:CD:EF:12:34:...
        X509v3 Authority Key Identifier:
            12:34:56:78:AB:...
    Signature Algorithm: sha256WithRSAEncryption
    Signature Value:
        30:82:01:22:ab:cd:ef:...  ← CA 서명
```

## 인증서 종류

### 검증 수준에 따른 분류

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 종류 (검증 수준)                        │
└─────────────────────────────────────────────────────────────┘

1. DV (Domain Validated) 인증서
   ┌─────────────────────────────────────────┐
   │  검증: 도메인 소유만 확인               │
   │  방법: 이메일, DNS 레코드, HTTP 파일    │
   │  시간: 수 분 ~ 수 시간                  │
   │  비용: 무료 ~ 저렴 (Let's Encrypt 등)   │
   │  용도: 개인 블로그, 소규모 웹사이트     │
   │  표시: 자물쇠 아이콘만                  │
   └─────────────────────────────────────────┘

2. OV (Organization Validated) 인증서
   ┌─────────────────────────────────────────┐
   │  검증: 도메인 + 조직 실사               │
   │  방법: 공문, 전화 확인, 사업자등록증    │
   │  시간: 1-3 영업일                       │
   │  비용: 연간 $100-$300                   │
   │  용도: 기업 웹사이트, 이커머스          │
   │  표시: 조직 정보 포함                   │
   └─────────────────────────────────────────┘

3. EV (Extended Validation) 인증서
   ┌─────────────────────────────────────────┐
   │  검증: 강화된 조직 실사                 │
   │  방법: 엄격한 신원 확인, 법적 문서     │
   │  시간: 3-7 영업일                       │
   │  비용: 연간 $300-$600                   │
   │  용도: 은행, 금융, 대규모 이커머스      │
   │  표시: 녹색 주소창 (과거), 조직명 명시  │
   └─────────────────────────────────────────┘

비교:
  ┌──────────────┬────────┬────────┬────────┐
  │  항목        │   DV   │   OV   │   EV   │
  ├──────────────┼────────┼────────┼────────┤
  │ 도메인 검증  │   ✓    │   ✓    │   ✓    │
  │ 조직 검증    │   ✗    │   ✓    │   ✓    │
  │ 법적 실사    │   ✗    │   ✗    │   ✓    │
  │ 발급 시간    │ 수분   │ 1-3 일 │ 3-7 일 │
  │ 비용         │ 무료~  │ $$     │ $$$    │
  │ 신뢰도       │ 보통   │ 높음   │ 매우높음│
  └──────────────┴────────┴────────┴────────┘
```

### 용도에 따른 분류

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 종류 (용도)                             │
└─────────────────────────────────────────────────────────────┘

1. 서버 인증서 (SSL/TLS)
   - 웹사이트 HTTPS 암호화
   - 예: www.google.com 인증서

2. 클라이언트 인증서
   - 사용자 인증용
   - 예: Kubernetes kubectl 인증서

3. 코드 서명 인증서
   - 소프트웨어 무결성 보장
   - 예: Windows 드라이버 서명

4. 이메일 인증서 (S/MIME)
   - 이메일 암호화 및 서명
   - 예: Outlook 서명 인증서

5. CA 인증서
   - 다른 인증서 서명용
   - 예: Root CA, Intermediate CA
```

## 인증서 생명주기

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 생명주기                                │
└─────────────────────────────────────────────────────────────┘

1. 생성 (Enrollment)
   │
   ▼
   키 쌍 생성 → CSR 생성 → CA 에 신청
   │
   ▼
2. 발급 (Issuance)
   │
   ▼
   신원 검증 → CA 서명 → 인증서 발급
   │
   ▼
3. 배포 (Deployment)
   │
   ▼
   웹 서버 설치 → 클라이언트에 제공
   │
   ▼
4. 사용 (Usage)
   │
   ▼
   HTTPS 통신 → 정기적 상태 확인 (OCSP)
   │
   ▼
5. 갱신 (Renewal)
   │
   ▼
   만료 전 재발급 → 새 인증서로 교체
   │
   ▼
6. 폐기 (Revocation) [필요시]
   │
   ▼
   개인키 유출 → CRL 등록 → OCSP 폐기 표시
   │
   ▼
7. 만료 (Expiration)
   │
   ▼
   자동 무효화 → 새 인증서 필요
```

### 인증서 유효기간

```
┌─────────────────────────────────────────────────────────────┐
│              인증서 유효기간 추이                           │
└─────────────────────────────────────────────────────────────┘

과거:
  - Root CA: 20-25 년
  - 서버 인증서: 5-10 년
```

## Kubernetes 와 X.509

### Kubernetes 에서 사용되는 X.509 인증서

```
┌─────────────────────────────────────────────────────────────┐
│              Kubernetes 의 X.509 인증서                     │
└─────────────────────────────────────────────────────────────┘

1. API Server 서버 인증서
   - 용도: HTTPS 서버
   - Subject: CN=kube-apiserver
   - SAN: kubernetes.default, 10.96.0.1, ...

2. kubelet 클라이언트 인증서
   - 용도: API Server 인증
   - Subject: CN=system:node:<노드이름>
   - O=system:nodes

3. controller-manager 클라이언트 인증서
   - 용도: API Server 인증
   - Subject: CN=system:kube-controller-manager
   - O=system:kube-controller-manager

4. scheduler 클라이언트 인증서
   - 용도: API Server 인증
   - Subject: CN=system:kube-scheduler
   - O=system:kube-scheduler

5. etcd 서버 인증서
   - 용도: etcd HTTPS 서버
   - Subject: CN=etcd-server

6. etcd 피어 인증서
   - 용도: etcd 간 통신
   - Subject: CN=etcd-peer

7. front-proxy 클라이언트 인증서
   - 용도: 애그리게이션 레이어
   - Subject: CN=front-proxy-client
```

### Kubernetes 인증서 특징

```
┌─────────────────────────────────────────────────────────────┐
│              Kubernetes 인증서 특징                         │
└─────────────────────────────────────────────────────────────┘

1. 자체 CA 운영
   - 외부 CA 사용 안 함
   - kubeadm 이 자동 생성
   - /etc/kubernetes/pki/ 에 저장

2. CN 과 O 를 RBAC 에 활용
   - CN: 사용자/컴포넌트 이름
   - O: 그룹 (RBAC 바인딩용)
   - 예: O=system:masters → cluster-admin 권한

3. 짧은 유효기간
   - 기본 1 년
   - kubelet 인증서: 자동 회전 (1 년마다)
   - 정기적 갱신 필요

4. 많은 인증서
   - 약 10-15 개 인증서 사용
   - 각 컴포넌트별 인증서
   - 관리 복잡
```

## 요약

```
┌─────────────────────────────────────────────────────────────┐
│                    핵심 요약                                │
├─────────────────────────────────────────────────────────────┤
│  1. PKI (공개키 기반구조)                                   │
│     - 공개키 암호화를 안전하게 사용하기 위한 인프라         │
│     - CA, RA, 인증서 저장소, CRL, OCSP 등으로 구성          │
│     - "누구의 공개키인가?"를 보증                           │
│                                                             │
│  2. X.509 인증서                                            │
│     - ITU-T 표준 디지털 인증서 포맷                         │
│     - 버전 3 (v3) 이 현재 표준                              │
│     - Issuer, Subject, 공개키, CA 서명 등 포함              │
│     - v3 확장: SAN, Key Usage, EKU 등                       │
│                                                             │
│  3. Distinguished Name (DN)                                 │
│     - CN (Common Name): 가장 중요 (도메인/이름)             │
│     - O (Organization): 조직명                              │
│     - C (Country): 국가 코드                                │
│                                                             │
│  4. 인증서 종류                                             │
│     - DV: 도메인만 검증 (무료~)                             │
│     - OV: 조직 검증 ($100-$300)                             │
│     - EV: 강화된 검증 ($300-$600)                           │
│                                                             │
│  5. 인증서 생명주기                                         │
│     - 생성 → 발급 → 배포 → 사용 → 갱신 → (폐기) → 만료      │
│     - 유효기간 점점 짧아짐 (현재 398 일)                     │
│                                                             │
│  6. Kubernetes 와 X.509                                     │
│     - 자체 CA 운영 (kubeadm 이 자동 생성)                   │
│     - CN 과 O 를 RBAC 에 활용                               │
│     - 약 10-15 개 인증서 사용                               │
│     - 기본 1 년 유효기간                                    │
└─────────────────────────────────────────────────────────────┘
```

**PKI 는 디지털 신뢰의 인프라이며, X.509 는 그 표준입니다. Kubernetes 도 이 표준을 따라 안전한 통신과 인증을 구현합니다.**
