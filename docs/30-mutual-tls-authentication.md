# Mutual TLS Authentication (mTLS)

상호 TLS 인증 (mTLS) 에 대해 알아봅니다.

---

## mTLS 란?

### 개념

```
┌─────────────────────────────────────────────────────────────┐
│          Mutual TLS Authentication (mTLS)                   │
└─────────────────────────────────────────────────────────────┘

정의:
  - 양방향 TLS 인증
  - 클라이언트와 서버가 서로 인증서로 신원 확인
  - 단방향 TLS 와 대비됨

단방향 TLS vs 양방향 TLS (mTLS):

단방향 TLS (일반 HTTPS):
  ┌─────────────┐                    ┌─────────────┐
  │  클라이언트  │                    │    서버     │
  │             │  1. Server Cert    │             │
  │             │◀───────────────────│             │
  │             │     검증           │             │
  │             │───────────────────▶│             │
  │             │  2. 암호화 통신     │             │
  │             │◀══════════════════▶│             │
  │             │                    │             │
  │  서버만 인증 │                    │  서버 인증서│
  └─────────────┘                    └─────────────┘

양방향 TLS (mTLS):
  ┌─────────────┐                    ┌─────────────┐
  │  클라이언트  │                    │    서버     │
  │             │  1. Server Cert    │             │
  │             │◀───────────────────│             │
  │             │     검증           │             │
  │  Client Cert│───────────────────▶│             │
  │             │     검증           │             │
  │             │◀───────────────────│             │
  │             │  2. 암호화 통신     │             │
  │             │◀══════════════════▶│             │
  │             │                    │             │
  │  양쪽 인증   │                    │  양쪽 인증  │
  └─────────────┘                    └─────────────┘
```

### mTLS 의 목적

```
┌─────────────────────────────────────────────────────────────┐
│          mTLS 의 목적                                       │
└─────────────────────────────────────────────────────────────┘

1. 상호 신원 확인
   - 서버: 클라이언트 신원 확인
   - 클라이언트: 서버 신원 확인
   - 양방향 신뢰 확립

2. 인증된 접근 제어
   - 인증서 가진 클라이언트만 접근 허용
   - 무단 접근 차단
   - 제로 트러스트 보안 모델

3. 보안 강화
   - 인증서 기반 강력한 인증
   - 비밀번호 불필요
   - 피싱 공격 방지
```

## mTLS 동작 원리

### mTLS 핸드셰이크 과정

```
┌─────────────────────────────────────────────────────────────┐
│          mTLS 핸드셰이크 과정                               │
└─────────────────────────────────────────────────────────────┘

클라이언트                              서버
   │                                      │
   │  1. ClientHello                      │
   │  (지원 TLS 버전, 암호화 스위트)       │
   │─────────────────────────────────────▶│
   │                                      │
   │  2. ServerHello                      │
   │  (선택된 TLS 버전, 암호화 스위트)     │
   │◀─────────────────────────────────────│
   │                                      │
   │  3. Certificate (서버 인증서)        │
   │  - 서버 인증서 전송                  │
   │  - CA 체인 포함                      │
   │◀─────────────────────────────────────│
   │                                      │
   │  4. Certificate Request              │
   │  - 클라이언트 인증서 요청            │
   │  - 허용된 CA 목록 전송               │
   │◀─────────────────────────────────────│
   │                                      │
   │  5. Certificate (클라이언트 인증서)  │
   │  - 클라이언트 인증서 전송            │
   │  - 클라이언트 개인키로 서명          │
   │─────────────────────────────────────▶│
   │                                      │
   │  6. Certificate Verify               │
   │  - 클라이언트 인증서 검증            │
   │  - CA 로 서명 확인                   │
   │─────────────────────────────────────▶│
   │                                      │
   │  7. Key Exchange                     │
   │  - Premaster Secret 암호화 전송      │
   │─────────────────────────────────────▶│
   │                                      │
   │  8. Finished (양쪽 모두)             │
   │  - 핸드셰이크 완료                   │
   │◀════════════════════════════════════▶│
   │                                      │
   │  9. 암호화 통신 시작                 │
   │◀════════════════════════════════════▶│
   └──────────────────────────────────────┘
```

### 단계별 상세 설명

```
┌─────────────────────────────────────────────────────────────┐
│          mTLS 핸드셰이크 단계별 설명                        │
└─────────────────────────────────────────────────────────────┘

Step 1-2: Hello 교환
  - TLS 버전, 암호화 스위트 협상
  - 양쪽 모두 지원 가능한 설정 확인

Step 3: 서버 인증서 전송
  - 서버가 자신의 인증서 전송
  - 클라이언트가 서버 인증서 검증
  - CA 로 서명 확인, 유효기간 확인

Step 4: Certificate Request
  - 서버가 클라이언트에게 인증서 요청
  - "어떤 CA 로부터 발급받은 인증서인가?" 목록 전송
  - mTLS 의 핵심 단계!

Step 5: 클라이언트 인증서 전송
  - 클라이언트가 자신의 인증서 전송
  - 서버가 허용한 CA 로부터 발급받은 인증서
  - 클라이언트 개인키로 서명 증명

Step 6: Certificate Verify
  - 클라이언트 인증서 검증
  - 서버가 CA 로 서명 확인
  - 인증서 유효기간 확인
  - 인증서 폐기 여부 확인 (CRL/OCSP)

Step 7-9: 키 교환 및 통신
  - 세션 키 생성
  - 암호화 통신 시작
```

## Kubernetes 와 mTLS

### Kubernetes 에서 mTLS 사용처

```
┌─────────────────────────────────────────────────────────────┐
│          Kubernetes 에서 mTLS 사용처                        │
└─────────────────────────────────────────────────────────────┘

1. API Server ↔ kubelet 통신
   ┌─────────────────────────────────────────┐
   │  API Server                             │
   │  - 서버 인증서: apiserver.crt           │
   │  - 클라이언트 인증서:                   │
   │    apiserver-kubelet-client.crt         │
   │                                         │
   │  kubelet                                │
   │  - 서버 인증서: kubelet.crt             │
   │  - 클라이언트 인증서: kubelet-client.crt│
   │                                         │
   │  양쪽 모두 상대 인증서 검증!            │
   └─────────────────────────────────────────┘

2. API Server ↔ etcd 통신
   ┌─────────────────────────────────────────┐
   │  API Server                             │
   │  - 클라이언트 인증서:                   │
   │    apiserver-etcd-client.crt            │
   │                                         │
   │  etcd                                   │
   │  - 서버 인증서: etcd/server.crt         │
   │  - 피어 인증서: etcd/peer.crt           │
   │                                         │
   │  양방향 인증으로 etcd 보호!             │
   └─────────────────────────────────────────┘

3. kubectl ↔ API Server 통신
   ┌─────────────────────────────────────────┐
   │  kubectl (클라이언트)                   │
   │  - 클라이언트 인증서: admin.crt         │
   │  - 클라이언트 개인키: admin.key         │
   │                                         │
   │  API Server (서버)                      │
   │  - 서버 인증서: apiserver.crt           │
   │                                         │
   │  kubectl 이 API Server 인증서 검증      │
   │  API Server 가 kubectl 인증서 검증      │
   └─────────────────────────────────────────┘
```

## mTLS 설정 예시

### API Server ↔ kubelet mTLS

```yaml
# API Server 설정 (kube-apiserver.yaml)
apiVersion: v1
kind: Pod
metadata:
  name: kube-apiserver
  namespace: kube-system
spec:
  containers:
  - name: kube-apiserver
    image: registry.k8s.io/kube-apiserver:v1.31.4
    command:
    - kube-apiserver
    
    # kubelet 클라이언트 인증서 (API Server → kubelet)
    - --kubelet-client-certificate=/etc/kubernetes/pki/apiserver-kubelet-client.crt
    - --kubelet-client-key=/etc/kubernetes/pki/apiserver-kubelet-client.key
    
    # kubelet 인증서 검증용 CA
    - --kubelet-certificate-authority=/etc/kubernetes/pki/ca.crt
```

```yaml
# kubelet 설정 (kubelet-config.yaml)
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration

# 서버 인증서 (kubelet → API Server)
- --tls-cert-file=/var/lib/kubelet/pki/kubelet.crt
- --tls-private-key-file=/var/lib/kubelet/pki/kubelet.key

# API Server CA 인증서 (kubelet 이 API Server 검증)
- --authorization-mode=Webhook
- --client-ca-file=/etc/kubernetes/pki/ca.crt
```

### kubectl mTLS 설정

```bash
# kubeconfig 에 클라이언트 인증서 설정
kubectl config set-credentials admin \
  --client-certificate=/root/.kube/admin.crt \
  --client-key=/root/.kube/admin.key

# kubeconfig 확인
kubectl config view --minify

# 출력:
# users:
# - name: admin
#   user:
#     client-certificate-data: LS0tLS1...
#     client-key-data: LS0tLS1...

# mTLS 통신으로 API Server 접속
kubectl get pods
# 1. kubectl 이 API Server 인증서 검증
# 2. API Server 가 kubectl 인증서 검증
# 3. 양방향 인증 완료 후 통신
```

## mTLS vs 단방향 TLS

### 비교 표

```
┌─────────────────────────────────────────────────────────────┐
│          mTLS vs 단방향 TLS                                 │
└─────────────────────────────────────────────────────────────┘

┌──────────────────────┬──────────────────────────────────────┐
│  mTLS (양방향)       │  단방향 TLS                          │
├──────────────────────┼──────────────────────────────────────┤
│  인증 방향           │  인증 방향                           │
│  클라이언트 ↔ 서버   │  클라이언트 → 서버                   │
├──────────────────────┼──────────────────────────────────────┤
│  서버 인증           │  서버 인증                           │
│  필요 (인증서)       │  필요 (인증서)                       │
├──────────────────────┼──────────────────────────────────────┤
│  클라이언트 인증     │  클라이언트 인증                     │
│  필요 (인증서)       │  불필요 (비밀번호 등)                │
├──────────────────────┼──────────────────────────────────────┤
│  보안성              │  보안성                              │
│  매우 높음           │  높음                                │
├──────────────────────┼──────────────────────────────────────┤
│  복잡도              │  복잡도                              │
│  높음 (양쪽 관리)    │  낮음 (서버만 관리)                  │
├──────────────────────┼──────────────────────────────────────┤
│  사용 사례           │  사용 사례                           │
│  - Kubernetes 내부   │  - 일반 웹사이트 (HTTPS)             │
│  - 마이크로서비스    │  - 일반 API                         │
│  - 제로 트러스트     │  - 공개 서비스                       │
├──────────────────────┼──────────────────────────────────────┤
│  성능                │  성능                                │
│  약간 느림 (검증 2 회)│  빠름 (검증 1 회)                   │
└──────────────────────┴──────────────────────────────────────┘
```

### 선택 가이드

```
┌─────────────────────────────────────────────────────────────┐
│          mTLS vs 단방향 TLS 선택 가이드                     │
└─────────────────────────────────────────────────────────────┘

mTLS 를 사용해야 하는 경우:
  ✓ 클러스터 내부 통신 (Kubernetes 컴포넌트)
  ✓ 마이크로서비스 간 통신
  ✓ 제로 트러스트 보안 모델
  ✓ 강력한 인증 필요
  ✓ 비밀번호 관리 어려움

단방향 TLS 를 사용하는 경우:
  ✓ 공개 웹사이트 (HTTPS)
  ✓ 일반 API 서비스
  ✓ 간단한 인증으로 충분
  ✓ 성능이 중요
  ✓ 인증서 관리 복잡도 피함
```

## mTLS 장점과 단점

### 장점

```
┌─────────────────────────────────────────────────────────────┐
│          mTLS 장점                                          │
└─────────────────────────────────────────────────────────────┘

1. 강력한 인증
   - 인증서 기반 신원 확인
   - 비밀번호보다 안전
   - 피싱 공격 방지

2. 상호 신뢰
   - 클라이언트도 서버 검증
   - 서버도 클라이언트 검증
   - 양방향 신뢰 확립

3. 접근 제어
   - 인증서 가진 클라이언트만 접근
   - 무단 접근 원천 차단
   - 세분화된 권한 부여 가능

4. 감사 추적
   - 인증서로 사용자 식별
   - 누가 접속했는지 추적
   - 감사 로그에 기록

5. 자동화 용이
   - 인증서 기반 자동 인증
   - 비밀번호 관리 불필요
   - CI/CD 파이프라인 통합
```

### 단점

```
┌─────────────────────────────────────────────────────────────┐
│          mTLS 단점                                          │
└─────────────────────────────────────────────────────────────┘

1. 복잡도
   - 양쪽 인증서 관리 필요
   - 인증서 갱신 복잡
   - 설정이 어려움

2. 성능 오버헤드
   - 양방향 검증으로 지연
   - 핸드셰이크 시간 증가
   - CPU 사용량 증가

3. 인증서 관리
   - 클라이언트 인증서 배포
   - 만료일 관리
   - 폐기된 인증서 관리

4. 디버깅 어려움
   - 양쪽 인증서 문제 확인
   - CA 체인 검증
   - 오류 원인 파악 어려움
```

## mTLS 문제 해결

### 일반적인 오류

```
┌─────────────────────────────────────────────────────────────┐
│          mTLS 일반적인 오류                                 │
└─────────────────────────────────────────────────────────────┘

1. "certificate signed by unknown authority"
   원인: CA 인증서를 신뢰할 수 없음
   해결: CA 인증서를 신뢰 저장소에 추가

2. "client certificate required"
   원인: 서버가 클라이언트 인증서 요구
   해결: 클라이언트 인증서 설정

3. "certificate has expired"
   원인: 인증서 만료
   해결: 인증서 갱신

4. "certificate not valid for requested host"
   원인: SAN 이 일치하지 않음
   해결: 올바른 SAN 을 가진 인증서 사용
```

### 디버깅 명령어

```bash
# 서버 인증서 확인
openssl s_client -connect api-server:6443 \
  -showcerts

# 클라이언트 인증서로 접속 테스트
openssl s_client -connect api-server:6443 \
  -cert client.crt \
  -key client.key \
  -CAfile ca.crt

# 인증서 체인 검증
openssl verify -CAfile ca.crt client.crt

# 인증서 만료일 확인
openssl x509 -in client.crt -noout -dates
```

```
│  4. mTLS vs 단방향 TLS                                      │
│     - mTLS: 양쪽 인증, 매우 높은 보안, 복잡               │
│     - 단방향: 서버만 인증, 높은 보안, 단순                │
│                                                             │
│  5. mTLS 장점                                               │
│     - 강력한 인증, 상호 신뢰, 접근 제어                    │
│     - 감사 추적, 자동화 용이                               │
│                                                             │
│  6. mTLS 단점                                               │
│     - 복잡도, 성능 오버헤드, 인증서 관리                   │
│     - 디버깅 어려움                                        │
│                                                             │
│  7. Kubernetes 에서 mTLS                                    │
│     - 모든 컴포넌트 간 통신이 mTLS 로 보호                 │
│     - 제로 트러스트 보안 모델의 핵심                       │
└─────────────────────────────────────────────────────────────┘
```

**mTLS 는 Kubernetes 클러스터 보안의 핵심입니다. 모든 컴포넌트 간 통신이 양방향 인증으로 보호되며, 제로 트러스트 보안 모델을 구현합니다.**
