# Keycloak: 오픈소스 IAM 솔루션

Keycloak 은 OIDC(OpenID Connect) 와 OAuth 2.0 표준을 가장 강력하게 구현한 오픈소스 Identity and Access Management (IAM) 솔루션입니다.

---

## Keycloak 이란?

### 핵심 개념

```
┌─────────────────────────────────────────────────────────────┐
│          Keycloak 이란?                                     │
└─────────────────────────────────────────────────────────────┘

정의:
  - 오픈소스 Identity and Access Management (IAM) 솔루션
  - OIDC 와 OAuth 2.0 표준을 강력하게 구현
  - Red Hat 에서 개발 및 유지보수

쉽게 말해:
  ┌─────────────────────────────────────────┐
  │  내 서비스가 직접 로그인 로직을         │
  │  짤 필요 없이                           │
  │                                         │
  │  "인증과 인가를 전담해 주는              │
  │   서버 소프트웨어"                       │
  │                                         │
  │  를 통째로 가져다 쓰는 것!              │
  └─────────────────────────────────────────┘

비유:
  - 아파트 단지 (서비스) 의 수문장 (Keycloak)
  - 주민 (사용자) 인증은 수문장이 전담
  - 각 세대 (서비스) 는 인증만 믿고 사용
```

## 1. Keycloak 의 핵심 역할

### 4 가지 주요 역할

```
┌─────────────────────────────────────────────────────────────┐
│          Keycloak 의 핵심 역할                              │
└─────────────────────────────────────────────────────────────┘

1. Identity Provider (IdP)
   ┌─────────────────────────────────────────┐
   │  - 사용자 계정 정보를 저장하고 관리     │
   │  - 로그인을 시도하는 유저가 누구인지    │
   │    확인                                 │
   │                                         │
   │  예시:                                  │
   │  - 사용자명/비밀번호 확인               │
   │  - 계정 생성/수정/삭제                  │
   │  - 세션 관리                            │
   └─────────────────────────────────────────┘

2. Federation (연동)
   ┌─────────────────────────────────────────┐
   │  - 기존 회사 시스템과 연동               │
   │  - LDAP, Active Directory (AD) 연결     │
   │  - 기존 계정 정보를 그대로 사용         │
   │                                         │
   │  예시:                                  │
   │  - 회사 AD 계정으로 로그인              │
   │  - 기존 LDAP 사용자 동기화              │
   │  - 계정 통합 관리                       │
   └─────────────────────────────────────────┘

3. Identity Brokering
   ┌─────────────────────────────────────────┐
   │  - 외부 소셜 로그인과 연결              │
   │  - "로그인 전달자" 역할                 │
   │                                         │
   │  지원 서비스:                           │
   │  - Google                               │
   │  - GitHub                               │
   │  - Facebook                             │
   │  - Microsoft                            │
   │  - LinkedIn                             │
   │  - 등...                                │
   └─────────────────────────────────────────┘

4. Single Sign-On (SSO)
   ┌─────────────────────────────────────────┐
   │  - 한 번만 로그인하면                   │
   │  - Keycloak 에 연결된 여러 앱을         │
   │    추가 로그인 없이 이용                │
   │                                         │
   │  예시:                                  │
   │  - A 서비스 로그인 → B 서비스도 자동 로그인 │
   │  - C 서비스도 자동 로그인               │
   │  - D 서비스도 자동 로그인               │
   │                                         │
   │  ┌───┐    ┌───┐    ┌───┐               │
   │  │ A │    │ B │    │ C │               │
   │  └─┬─┘    └─┬─┘    └─┬─┘               │
   │    │        │        │                  │
   │    └────────┼────────┘                  │
   │             │                           │
   │          ┌──▼──┐                        │
   │          │Keycloak│                     │
   │          └──────┘                       │
   └─────────────────────────────────────────┘
```

### Keycloak 아키텍처

```
┌─────────────────────────────────────────────────────────────┐
│          Keycloak 아키텍처                                   │
└─────────────────────────────────────────────────────────────┘

                    ┌─────────────┐
                    │   사용자    │
                    └──────┬──────┘
                           │
                           │ 1. 로그인 요청
                           ▼
                    ┌─────────────┐
                    │  Keycloak   │
                    │             │
                    │  - 인증     │
                    │  - 인가     │
                    │  - 토큰발급 │
                    └──────┬──────┘
                           │
         ┌─────────────────┼─────────────────┐
         │                 │                 │
         ▼                 ▼                 ▼
   ┌───────────┐   ┌───────────┐   ┌───────────┐
   │  서비스 A  │   │  서비스 B  │   │  서비스 C  │
   │           │   │           │   │           │
   │  토큰검증  │   │  토큰검증  │   │  토큰검증  │
   └───────────┘   └───────────┘   └───────────┘

특징:
  - 중앙 집중식 인증 (Keycloak 이 전담)
  - 각 서비스는 토큰 검증만 수행
  - 한 번 로그인으로 모든 서비스 접근 (SSO)
```

## 3. 왜 Keycloak 을 쓰는가?

### 장단점

```
┌─────────────────────────────────────────────────────────────┐
│          Keycloak 장단점                                     │
└─────────────────────────────────────────────────────────────┘

장점 (Pros):
┌─────────────────────────────────────────────────────────────┐
│  1. 비용 절감                                               │
│     - 오픈소스이므로 라이선스 비용 없음                     │
│     - 엔터프라이즈급 보안 기능을 무료로                     │
│     - Okta, Auth0 의 유료 기능을 무료로 구현 가능           │
│                                                             │
│  2. 데이터 주권                                             │
│     - 클라우드 서비스 (SaaS) 와 달리                        │
│     - 온프레미스 (자체 서버) 설치 가능                      │
│     - 개인 클라우드에도 설치 가능                           │
│     - 민감한 사용자 정보를 직접 통제                        │
│                                                             │
│  3. 커스터마이징                                            │
│     - 로그인 화면 UI 개발자 마음대로 확장                   │
│     - 인증 로직 (SPI) 커스터마이징                          │
│     - 테마 (Theme) 변경으로 브랜드 일치                     │
│                                                             │
│  4. 표준 프로토콜                                           │
│     - OIDC, OAuth 2.0, SAML 2.0 지원                        │
│     - 다양한 서비스와 연동 가능                             │
│     - 벤더 락인 최소화                                      │
└─────────────────────────────────────────────────────────────┘

단점 (Cons):
┌─────────────────────────────────────────────────────────────┐
│  1. 운영 부담                                               │
│     - 서버 설치 직접 수행                                   │
│     - DB 연결 직접 설정                                     │
│     - 보안 패치 직접 적용                                   │
│     - 고가용성 (HA) 구성 직접 구축                          │
│     - (AWS Cognito 같은 SaaS 는 이 부분을 알아서 해줌)       │
│                                                             │
│  2. 학습 곡선                                               │
│     - 설정할 수 있는 옵션이 매우 방대                       │
│     - 초기 제대로 세팅하는 데 시간 걸림                     │
│     - Realm, Client, Role 등 개념 이해 필요                 │
│                                                             │
│  3. 리소스 요구                                             │
│     - 자체 서버 리소스 필요                                 │
│     - 데이터베이스 별도 필요                                │
│     - 모니터링/백업 시스템 직접 구축                        │
└─────────────────────────────────────────────────────────────┘
```

### Keycloak vs SaaS 비교

```
┌─────────────────────────────────────────────────────────────┐
│          Keycloak vs SaaS (Okta, Auth0, AWS Cognito)        │
└─────────────────────────────────────────────────────────────┘

┌──────────────────┬──────────────────────────────────────────┐
│  Keycloak        │  SaaS (Okta, Auth0, Cognito)             │
├──────────────────┼──────────────────────────────────────────┤
│  라이선스: 무료  │  라이선스: 유료 (사용자 수 과금)         │
├──────────────────┼──────────────────────────────────────────┤
│  설치: 직접      │  설치: 불필요 (클라우드)                 │
├──────────────────┼──────────────────────────────────────────┤
│  운영: 직접      │  운영: 제공사 담당                       │
├──────────────────┼──────────────────────────────────────────┤
│  패치: 직접      │  패치: 자동                              │
├──────────────────┼──────────────────────────────────────────┤
│  데이터: 내 서버 │  데이터: 제공사 클라우드                 │
├──────────────────┼──────────────────────────────────────────┤
│  커스터마이징:   │  커스터마이징: 제한적                    │
│  매우 자유로움   │                                          │
├──────────────────┼──────────────────────────────────────────┤
│  적합:           │  적합:                                   │
│  - 데이터 주권   │  - 빠른 도입                           │
│  - 비용 절감     │  - 운영 부담 최소화                      │
│  - 높은 커스터마이│  - 소규모 프로젝트                       │
│    즈 필요       │                                          │
│  - 대규모        │                                          │
└──────────────────┴──────────────────────────────────────────┘
```

## 4. 실무 활용 시나리오

### Keycloak 을 사용한 로그인 흐름

```
┌─────────────────────────────────────────────────────────────┐
│          Keycloak 실무 활용 시나리오                        │
└─────────────────────────────────────────────────────────────┘

선생님께서 IT 감사나 시스템 설계를 하신다면
다음과 같은 구조로 활용하게 됩니다:

1. 사용자: 앱의 "로그인" 버튼 클릭
   ┌─────────────┐
   │   사용자    │
   │  (홍길동)   │
   └──────┬──────┘
          │ 클릭
          ▼
2. 앱: 사용자를 Keycloak 로그인 페이지로 리다이렉트
   ┌─────────────┐
   │     앱      │
   └──────┬──────┘
          │ 리다이렉트
          ▼
   ┌─────────────┐
   │  Keycloak   │
   │  로그인페이지│
   └──────┬──────┘
          │ ID/PW 입력
          ▼
3. Keycloak: ID/PW 확인 후 ID Token & Access Token 발행
   ┌─────────────┐
   │  Keycloak   │
   │             │
   │  - ID 검증  │
   │  - 토큰발급 │
   └──────┬──────┘
          │ 토큰 전달
          ▼
4. 앱: 전달받은 토큰을 검증하고 서비스 이용 허가
   ┌─────────────┐
   │     앱      │
   │             │
   │  - 토큰검증 │
   │  - 서비스  │
   │    허용    │
   └─────────────┘

결과:
  ✓ 앱은 로그인 로직 짤 필요 없음
  ✓ Keycloak 이 인증 전담
  ✓ 앱은 토큰 검증만 하면 됨
```

## 5. Austin 기반 테크 환경에서의 입지

### 엔터프라이즈 환경에서 Keycloak

```
┌─────────────────────────────────────────────────────────────┐
│          Austin 기반 테크 환경에서의 Keycloak               │
└─────────────────────────────────────────────────────────────┘

Austin 지역의 많은 엔터프라이즈 환경에서는:

보안 규정 (Compliance) 문제:
  ┌─────────────────────────────────────────┐
  │  사용자 정보를 외부 SaaS 에             │
  │  맡기지 못하는 경우 많음                │
  │                                         │
  │  이유:                                  │
  │  - 금융 규정 (PCI-DSS)                 │
  │  - 의료 규정 (HIPAA)                   │
  │  - 정부 규정 (FedRAMP)                 │
  │  - 데이터 주권 요구                    │
  └─────────────────────────────────────────┘

해결책: Red Hat Build of Keycloak
  ┌─────────────────────────────────────────┐
  │  - 상용 버전 (유료 지원)                │
  │  - Red Hat 이 공식 지원                 │
  │  - 엔터프라이즈 기능 추가               │
  │  - 보안 패치 보장                       │
  │  - SLA (서비스 레벨 계약) 제공          │
  └─────────────────────────────────────────┘

사용 사례:
  - 금융기관: 고객 데이터 외부 위탁 불가
  - 의료기관: HIPAA 준수 필요
  - 정부기관: 데이터 주권 요구
```
