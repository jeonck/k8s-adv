# 대칭키와 비대칭키 - Kubernetes 인증 이해를 위한 기초

Kubernetes 의 인증 방식을 이해하기 위해 암호화의 기본 개념인 대칭키와 비대칭키에 대해 알아봅니다.

## 암호화 기본 개념

```
평문 (Plaintext) → [암호화] → 암호문 (Ciphertext)
암호문 (Ciphertext) → [복호화] → 평문 (Plaintext)
```

- **평문**: 암호화되지 않은 원본 데이터
- **암호문**: 암호화된 데이터
- **키 (Key)**: 암호화/복호화에 사용되는 비밀 값

## 대칭키 암호화 (Symmetric Key Encryption)

### 개념

```
┌─────────────────────────────────────────────────────────────┐
│              대칭키 암호화 방식                             │
└─────────────────────────────────────────────────────────────┘

      송신자                          수신자
    ┌─────────┐                    ┌─────────┐
    │  평문   │                    │  평문   │
    └────┬────┘                    └────▲────┘
         │                              │
         │  암호화                      │ 복호화
         ▼                              │
    ┌─────────┐     암호문      ┌──────┴──────┐
    │  암호   │────────────────▶│    암호     │
    │  (Encrypt)│               │   (Decrypt) │
    └────┬────┘               └──────▲───────┘
         │                           │
         │      ┌────────────┐       │
         └─────▶│  대칭키    │◀──────┘
                │  (Secret)  │
                └────────────┘

핵심: 암호화와 복호화에 동일한 키 사용
```

### 특징

| 특성 | 설명 |
|------|------|
| **키 개수** | 1 개 (송신자와 수신자가 공유) |
| **속도** | 빠름 (하드웨어 가속 용이) |
| **키 배포** | 어려움 (보안 채널 필요) |
| **용도** | 대량 데이터 암호화 |

### 대표 알고리즘

| 알고리즘 | 키 길이 | 설명 |
|----------|---------|------|
| **AES** | 128, 192, 256 비트 | 현재 가장 널리 사용 (권장) |
| **DES** | 56 비트 | 구식 (취약, 사용 금지) |
| **3DES** | 168 비트 | DES 개선 (점차 퇴출) |
| **ChaCha20** | 256 비트 | AES 대안 (소프트웨어 최적화) |

### 예시: AES-256

```python
# 의사코드 예시
key = "0123456789abcdef0123456789abcdef"  # 256 비트 (32 바이트)
plaintext = "Hello, Kubernetes!"

# 암호화
ciphertext = AES_Encrypt(key, plaintext)
# 결과: "a1b2c3d4e5f6..." (암호문)

# 복호화
decrypted = AES_Decrypt(key, ciphertext)
# 결과: "Hello, Kubernetes!" (원본 복원)
```

### 대칭키의 문제점: 키 배포

```
┌─────────────────────────────────────────────────────────────┐
│              대칭키 키 배포 문제                            │
└─────────────────────────────────────────────────────────────┘

Alice                         Bob
  │                           │
  │  키를 어떻게 전달할까?    │
  │                           │
  │  ┌─────────┐              │
  │  │  대칭키 │──────────────┼── ?
  │  └─────────┘              │
  │                           │
  │  문제:                      │
  │  1. 키를 보내다가 탈취당하면? │
  │  2. 처음 만나는 사람과 어떻게 키 공유? │
  │  3. n 명이 서로 통신하려면? │
  │     → n(n-1)/2 개의 키 필요 │
  └───────────────────────────┘
```

### 키 배포 문제의 본질

```
┌─────────────────────────────────────────────────────────────┐
│          대칭키의 근본적인 문제: "키 배송의 역설"           │
└─────────────────────────────────────────────────────────────┘

상황: Alice 가 Bob 에게 비밀 메시지를 보내고 싶음

문제:
1. 메시지를 암호화하려면 공통의 키가 필요함
2. 하지만 키를 안전하게 전달할 방법이 없음
3. 키를 보내다가 해커에게 탈취당하면?
4. 암호화된 메시지 + 키 = 모두 노출!

역설:
"보안 통신을 하려면 안전한 키 배송이 필요
 그런데 안전한 키 배송을 하려면 이미 보안 통신이 필요?"

마치:
"금고를 열려면 열쇠가 필요
 그런데 열쇠를 보내려면 금고가 필요?"
```

### 키 배포 시나리오

```
시나리오 1: 직접 전달 (오프라인)
┌─────────────────────────────────────────────────────────────┐
│  Alice 가 USB 에 키를 담아 Bob 에게 직접 전달               │
│                                                             │
│  ✓ 안전함                                                  │
│  ✗ 비실용적 (먼 거리, 많은 사용자)                         │
│  ✗ 확장성 없음                                             │
└─────────────────────────────────────────────────────────────┘

시나리오 2: 암호화된 채널로 전송
┌─────────────────────────────────────────────────────────────┐
│  Alice 가 키를 암호화해서 Bob 에게 전송                    │
│                                                             │
│  ✓ 편리함                                                  │
│  ✗ 그 암호화에 사용할 키는 어떻게 전달? (무한 루프)        │
│  ✗ "키를 위한 키" 문제                                     │
└─────────────────────────────────────────────────────────────┘

시나리오 3: 제 3 자에게 의존
┌─────────────────────────────────────────────────────────────┐
│  신뢰할 수 있는 제 3 자 (Courier) 가 키 전달               │
│                                                             │
│  ✓ 실용적                                                  │
│  ✗ 제 3 자를 완전히 신뢰해야 함                            │
│  ✗ 제 3 자가 해킹당하면 모든 키 유출                       │
└─────────────────────────────────────────────────────────────┘
```

### 키 관리 복잡성

```
2 명 통신: 1 개 키 필요
3 명 통신: 3 개 키 필요
10 명 통신: 45 개 키 필요
100 명 통신: 4,950 개 키 필요

n 명이 상호 통신할 때 필요한 키 수: n(n-1)/2

┌─────────────────────────────────────────────────────────────┐
│              사용자 수 증가에 따른 키 개수                  │
│                                                             │
│   100 사용자 → 4,950 개 키 관리                            │
│   1,000 사용자 → 499,500 개 키 관리                        │
│   10,000 사용자 → 49,995,000 개 키 관리                    │
│                                                             │
│  현실적으로 관리 불가능!                                    │
└─────────────────────────────────────────────────────────────┘
```

## 비대칭키가 해결한 문제: 키 배포

### 대칭키 문제의 해결사

```
┌─────────────────────────────────────────────────────────────┐
│          비대칭키가 가져온 혁명: 키 배포 문제 해결          │
└─────────────────────────────────────────────────────────────┘

대칭키 시대:
  "키를 안전하게 전달할 방법이 없다!"
  → 보안 통신의 근본적 한계

비대칭키 등장 (1976 년, Diffie-Hellman):
  "암호화와 복호화에 다른 키를 쓰면 된다!"
  → 공개키는 모두에게, 개인키는 나만
  → 키 배포 문제 해결!
```

### 비대칭키의 핵심 아이디어

```
┌─────────────────────────────────────────────────────────────┐
│              비대칭키 키 배포 방식                          │
└─────────────────────────────────────────────────────────────┘

Alice                         Bob
  │                           │
  │  1. Bob 이 키 쌍 생성     │
  │     (공개키 + 개인키)     │
  │                           │
  │  2. Bob 이 공개키 배포    │
  │     (웹사이트, 디렉토리)  │
  │     ┌─────────┐           │
  │     │공개키   │───────────┼──▶ 누구나 접근 가능
  │     └─────────┘           │
  │                           │
  │  3. Alice 가 공개키로     │
  │     메시지 암호화         │
  │     ┌─────────┐           │
  │     │암호화   │           │
  │     └────┬────┘           │
  │          │                │
  │          │────────────────┼──▶
  │                           │
  │  4. Bob 이 개인키로       │
  │     복호화 (나만 가능)    │
  │                           │
  │  핵심:                      │
  │  - 공개키는 유출되어도 괜찮음 │
  │  - 개인키는 절대 공유 안 함 │
  │  - 키 배포 문제 해결!     │
  └───────────────────────────────┘
```

### 공개키 배포 방법

```
방법 1: 웹사이트 게시
┌─────────────────────────────────────────────────────────────┐
│  Bob 의 웹사이트: https://bob.com/public-key.pem            │
│                                                             │
│  Alice 는 언제든지 다운로드 가능                            │
│  해커가 봐도 괜찮음 (공개키이므로)                          │
└─────────────────────────────────────────────────────────────┘

방법 2: 공개 키 디렉토리
┌─────────────────────────────────────────────────────────────┐
│  키 서버: keys.openpgp.org                                  │
│                                                             │
│  search: bob@example.com                                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Bob Smith <bob@example.com>                         │   │
│  │ pub   rsa4096 2024-01-01                            │   │
│  │ Fingerprint: ABCD 1234 5678 ...                     │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  누구나 검색하고 다운로드 가능                              │
└─────────────────────────────────────────────────────────────┘

방법 3: 인증서 (CA 발급)
┌─────────────────────────────────────────────────────────────┐
│  신뢰할 수 있는 CA 가 Bob 의 공개키에 서명                  │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │            디지털 인증서                             │   │
│  │  소유자: Bob Smith                                   │   │
│  │  공개키: RSA 2048 bit                                │   │
│  │  발급자: DigiCert CA                                 │   │
│  │  유효기간: 2024-01-01 ~ 2025-01-01                  │   │
│  │  서명: (CA 의 디지털 서명)                           │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  CA 를 신뢰하면 공개키도 신뢰할 수 있음                     │
└─────────────────────────────────────────────────────────────┘
```

### 비대칭키의 장점

```
┌─────────────────────────────────────────────────────────────┐
│              비대칭키의 장점                                │
└─────────────────────────────────────────────────────────────┘

1. 키 배포가 쉽다
   - 공개키는 누구나 알 수 있음
   - 웹사이트, 이메일, 명함에 게시 가능
   - 유출되어도 안전 (개인키는 따로 보관)

2. 사전 공유가 불필요
   - 처음 만나는 사람과도 보안 통신 가능
   - 인터넷 쇼핑몰 (첫 방문) 과 HTTPS 통신
   - 낯선 웹사이트와도 안전한 결제

3. 키 관리가 간단
   - n 명이 통신할 때 필요한 키: n 개 (각자 키 쌍)
   - 대칭키: n(n-1)/2 개
   - 100 명: 100 개 vs 4,950 개

4. 디지털 서명 가능
   - 개인키로 서명 → 공개키로 검증
   - 부인 방지 (non-repudiation)
   - 문서 위변조 방지

5. 확장성
   - 새로운 사용자 추가가 쉬움
   - 기존 사용자와 키 재교환 불필요
   - 인터넷 규모의 통신 가능
```

### 비교: 대칭키 vs 비대칭키 키 관리

```
┌─────────────────────────────────────────────────────────────┐
│          100 사용자 환경에서 키 관리 비교                   │
└─────────────────────────────────────────────────────────────┘

대칭키:
  필요한 키 수: 4,950 개
  
  관리 작업:
  - 4,950 개 키 생성
  - 4,950 개 키 안전하게 배포
  - 4,950 개 키 저장소 관리
  - 키 순환 시 4,950 개 모두 교체
  - 현실적으로 불가능!

비대칭키:
  필요한 키 수: 100 개 (각자 1 개씩 키 쌍)
  
  관리 작업:
  - 각자가 자신의 키 쌍 생성
  - 공개키는 웹사이트에 게시
  - 개인키는 각자 보관
  - 키 순환 시 각자 교체
  - 매우 실용적!
```

### 비대칭키의 단점

```
┌─────────────────────────────────────────────────────────────┐
│              비대칭키의 단점                                │
└─────────────────────────────────────────────────────────────┘

1. 느림
   - 대칭키보다 100-1000 배 느림
   - 대량 데이터 암호화에 부적합
   - 예: RSA-2048 암호화 속도 ≈ 1 MB/s
         AES-256 암호화 속도 ≈ 1 GB/s

2. 긴 키 길이
   - RSA-2048: 256 바이트
   - AES-256: 32 바이트
   - 동일한 보안 수준에 8 배 더 긴 키

3. 개인키 관리 부담
   - 개인키 분실 = 복호화 불가
   - 개인키 유출 = 모든 통신 노출
   - 백업과 보안의 균형 필요

4. 양자 컴퓨터 위협
   - 쇼어 알고리즘 (Shor's algorithm)
   - RSA, ECC 를 다 깰 수 있음
   - 양자내성암호 (PQC) 개발 중
```

### 결론: 왜 비대칭키가 필요한가?

```
┌─────────────────────────────────────────────────────────────┐
│          비대칭키의 필요성: 요약                            │
└─────────────────────────────────────────────────────────────┘

질문: "대칭키만 쓰면 안 되나요?"

답변: "키 배포 문제가 해결되지 않습니다"

대칭키만 있는 세상:
  - 인터넷 쇼핑몰에서 카드 정보 입력 불가
  - 첫 방문 웹사이트와 보안 통신 불가
  - 은행 앱 사용 불가
  - HTTPS 자체가 불가능
  - 현대 인터넷은 존재할 수 없음

비대칭키가 해결한 것:
  ✓ 모르는 사람과도 보안 통신
  ✓ 안전한 키 교환
  ✓ 디지털 서명과 인증
  ✓ 확장 가능한 키 관리
  ✓ 현대 인터넷의 기반

그래서:
  비대칭키 = 현대 암호화의 혁명
  (1976 년 Diffie-Hellman 이후 모든 것이 변함)
```

## 비대칭키 암호화 (Asymmetric Key Encryption)

### 개념

```
┌─────────────────────────────────────────────────────────────┐
│              비대칭키 암호화 방식                           │
└─────────────────────────────────────────────────────────────┘

      송신자                          수신자
    ┌─────────┐                    ┌─────────┐
    │  평문   │                    │  평문   │
    └────┬────┘                    └────▲────┘
         │                              │
         │  암호화                      │ 복호화
         ▼                              │
    ┌─────────┐     암호문      ┌──────┴──────┐
    │  암호   │────────────────▶│    암호     │
    │  (Encrypt)│               │   (Decrypt) │
    └────┬────┘               └──────▲───────┘
         │                           │
         │      ┌────────────┐       │
         └─────▶│  공개키    │       │
                │  (Public)  │       │
                └────────────┘       │
                                     │
                ┌────────────┐       │
                │  개인키    │◀──────┘
                │  (Private) │
                └────────────┘
                   (수신자만 보유)

핵심: 암호화 (공개키) 와 복호화 (개인키) 에 서로 다른 키 사용
```

### 키 쌍 (Key Pair)

```
┌─────────────────────────────────────────────────────────────┐
│              공개키와 개인키 관계                           │
└─────────────────────────────────────────────────────────────┘

          ┌─────────────────────────┐
          │   키 쌍 (Key Pair)      │
          │   (수학적으로 연결됨)   │
          └────────────┬────────────┘
                       │
         ┌─────────────┴─────────────┐
         │                           │
         ▼                           ▼
    ┌─────────┐                ┌─────────┐
    │ 공개키  │                │ 개인키  │
    │ Public  │                │ Private │
    │  Key    │                │  Key    │
    └─────────┘                └─────────┘
         │                           │
         │ 누구나 알 수 있음         │ 절대 공개 금지
         │ 웹사이트, 디렉토리 등에   │ 소유자만 보관
         │ 게시 가능                │
```

### 두 가지 사용 방식

#### 1. 기밀성 (Confidentiality) - 암호화

```
┌─────────────────────────────────────────────────────────────┐
│              비대칭키 암호화 (기밀성)                       │
└─────────────────────────────────────────────────────────────┘

Alice                         Bob
  │                           │
  │  1. Bob 의 공개키를 얻음   │
  │  (웹사이트, 디렉토리)     │
  │                           │
  │  ┌─────────┐              │
  │  │  평문   │              │
  │  └────┬────┘              │
  │       │                   │
  │       │ 암호화            │
  │       ▼                   │
  │  ┌─────────┐              │
  │  │암호화문 │──────────────┼──▶
  │  └─────────┘   Bob 의      │
  │              공개키로 암호화│
  │                           │ 2. 개인키로 복호화
  │                           │    (Bob 만 가능)
  │                           ▼
  │                      ┌─────────┐
  │                      │  평문   │
  │                      └─────────┘
  │
  │  결과: Bob 만 읽을 수 있는 메시지
  └───────────────────────────────┘
```

#### 2. 인증 (Authentication) - 디지털 서명

```
┌─────────────────────────────────────────────────────────────┐
│              비대칭키 서명 (인증)                           │
└─────────────────────────────────────────────────────────────┘

Alice                         Bob
  │                           │
  │  1. 평문을 개인키로 서명   │
  │  ┌─────────┐              │
  │  │  평문   │              │
  │  └────┬────┘              │
  │       │                   │
  │       │ 서명              │
  │       ▼                   │
  │  ┌─────────┐              │
  │  │  서명   │──────────────┼──▶
  │  └─────────┘  Alice 의     │
  │              개인키로 서명  │
  │                           │ 2. 공개키로 검증
  │                           │    (모두 가능)
  │                           ▼
  │                      ┌─────────┐
  │                      │  검증   │
  │                      │  성공   │
  │                      └─────────┘
  │
  │  결과: Alice 가 보낸 것이 확실함
  └───────────────────────────────┘
```

### 대표 알고리즘

| 알고리즘 | 키 길이 | 설명 |
|----------|---------|------|
| **RSA** | 2048, 3072, 4096 비트 | 가장 널리 사용 |
| **ECDSA** | 256, 384 비트 | 타원곡선 기반 (짧은 키, 강력함) |
| **Ed25519** | 256 비트 | 고속 서명 (SSH, Git 에서 사용) |
| **Diffie-Hellman** | 2048+ 비트 | 키 교환 전용 |

### 예시: RSA-2048

```python
# 의사코드 예시

# 1. 키 쌍 생성
public_key, private_key = RSA_Generate_Key(2048)

# 2. 공개키는 모두에게 배포
publish(public_key)

# 3. 암호화 (기밀성)
ciphertext = RSA_Encrypt(public_key, "Secret Message")
# → 개인키를 가진 사람만 복호화 가능

# 4. 서명 (인증)
signature = RSA_Sign(private_key, "Document")
# → 공개키로 서명자 확인 가능
```

## 대칭키 vs 비대칭키 비교

```
┌─────────────────────────────────────────────────────────────┐
│           대칭키 vs 비대칭키 비교                           │
├─────────────────────────────────────────────────────────────┤
│  특성          │  대칭키      │  비대칭키                   │
├────────────────┼──────────────┼─────────────────────────────┤
│  키 개수       │  1 개         │  2 개 (공개키 + 개인키)      │
│  속도          │  빠름        │  느림 (100-1000 배 차이)    │
│  키 배포       │  어려움      │  쉬움 (공개키만 공유)       │
│  용도          │  데이터 암호화│  키 교환, 서명, 인증       │
│  알고리즘      │  AES, ChaCha │  RSA, ECDSA, Ed25519       │
│  키 길이       │  128-256 비트│  2048-4096 비트 (RSA)       │
│  보안성        │  키 관리 중요 │  개인키 관리 중요           │
└─────────────────────────────────────────────────────────────┘
```

## 하이브리드 방식: TLS/SSL

현대 암호화 통신은 대칭키와 비대칭키의 장점을 모두 사용합니다.

### TLS 핸드셰이크 흐름

```
┌─────────────────────────────────────────────────────────────┐
│              TLS 핸드셰이크 (간략화)                        │
└─────────────────────────────────────────────────────────────┘

클라이언트                          서버
    │                               │
    │  1. ClientHello               │
    │  (지원 암호화 스위트 목록)     │
    │──────────────────────────────▶│
    │                               │
    │  2. ServerHello + 인증서      │
    │  (서버 공개키 포함)           │
    │◀──────────────────────────────│
    │                               │
    │  3. 인증서 검증               │
    │  (CA 인증서로 서명 확인)       │
    │                               │
    │  4. 프리마스터 시크릿 생성    │
    │  (대칭키용 랜덤 값)           │
    │  ┌───────────────┐            │
    │  │프리마스터 시크릿│            │
    │  └───────┬───────┘            │
    │          │                    │
    │          │ 서버 공개키로 암호화│
    │          ▼                    │
    │  ┌───────────────┐            │
    │  │  암호화됨     │────────────┼──▶
    │  └───────────────┘            │
    │                               │ 5. 개인키로 복호화
    │                               │    프리마스터 시크릿 획득
    │                               │
    │  6. 세션 키 생성              │
    │  (대칭키, 양쪽 동일)          │
    │  ┌───────────────┐            │
    │  │  AES-256 키   │◀───────────┼──▶│  AES-256 키  │
    │  └───────────────┘            │
    │                               │
    │  7. 대칭키 암호화 통신 시작   │
    │  ─════════════════════════════│
    │  (빠른 대칭키 암호화)         │
    │                               │
    └───────────────────────────────┘
```

### 왜 하이브리드인가?

```
┌─────────────────────────────────────────────────────────────┐
│              하이브리드 방식의 장점                         │
└─────────────────────────────────────────────────────────────┘

1. 키 배포 문제 해결
   - 비대칭키로 안전한 키 교환
   - 처음 만나는 상대방과도 보안 통신 가능

2. 성능 최적화
   - 핸드셰이크: 비대칭키 (1 회)
   - 데이터 전송: 대칭키 (지속적)
   - AES 는 RSA 보다 1000 배 빠름

3. 완벽한 순방향 비밀성 (PFS)
   - 세션마다 새로운 대칭키 사용
   - 개인키 유출되도 과거 세션 안전
```

## Kubernetes 와의 연관성

### Kubernetes 에서의 사용 사례

```
┌─────────────────────────────────────────────────────────────┐
│          Kubernetes 의 암호화 사용처                        │
└─────────────────────────────────────────────────────────────┘

1. API Server 인증서 (TLS)
   ├─ 서버 인증서: 비대칭키 (RSA/ECDSA)
   ├─ 클라이언트 인증서: 비대칭키
   └─ 통신 암호화: 대칭키 (AES)

2. etcd 암호화
   ├─ 피어 간 통신: TLS (하이브리드)
   └─ 데이터 암호화: 대칭키 (AES-GCM)

3. ServiceAccount 토큰
   ├─ 토큰 서명: 비대칭키 (RSA/ECDSA)
   └─ 토큰 검증: 공개키

4. Secret 암호화
   ├─ etcd 저장 시: 대칭키 (AES)
   └─ 키 관리: KMS (비대칭키)

5. 네트워크 암호화
   ├─ CNI 플러그인 (Calico, Cilium)
   └─ IPSec/WireGuard (하이브리드)
```

### kubectl 접속 예시

```bash
# kubectl 이 API Server 에 접속할 때:

1. TLS 핸드셰이크
   ├─ 서버 인증서 검증 (비대칭키)
   └─ 세션 키 교환 (비대칭키 → 대칭키)

2. 클라이언트 인증
   ├─ 클라이언트 인증서 (비대칭키)
   └─ CN/O 에서 사용자 추출

3. 데이터 통신
   └─ AES-256 대칭키 암호화
```

## 용어 정리

| 용어 | 설명 |
|------|------|
| **평문 (Plaintext)** | 암호화되지 않은 원본 데이터 |
| **암호문 (Ciphertext)** | 암호화된 데이터 |
| **대칭키 (Symmetric Key)** | 암호화와 복호화에 같은 키 사용 |
| **공개키 (Public Key)** | 누구나 알 수 있는 키 (암호화/검증용) |
| **개인키 (Private Key)** | 소유자만 아는 키 (복호화/서명용) |
| **키 쌍 (Key Pair)** | 공개키와 개인키의 쌍 |
| **디지털 서명** | 개인키로 서명, 공개키로 검증 |
| **인증서 (Certificate)** | 공개키에 CA 가 서명한 것 |
| **CA (Certificate Authority)** | 인증서를 발급하는 신뢰 기관 |
| **TLS/SSL** | 보안 통신 프로토콜 (하이브리드 방식) |
